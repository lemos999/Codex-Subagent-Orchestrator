# Rex Baek (Opus) — 운영 관점 결과 (원본, 코드 분석 기반)

## 핵심 진단
설계 문서 8개, /persona 스킬(15명령+3엔진), 대시보드 v2, 서버 API 9개, JSON 저장소가 존재하나, 이 모든 것은 **"사람이 명령할 때만 움직이는 시스템"**. 24시간 살아있는 국가가 되려면 다음 4개가 빠져 있다.

## 발견 4개

### 1. 틱 데몬 (Tick Daemon) — 가장 치명적인 빈 곳
설계서에 "30분마다 1틱, 각 페르소나가 1개 행동을 선택·실행"이라 명시했으나, **이걸 구동하는 주체가 없다.** 현재 시뮬레이션은 `/persona` 명령을 사람이 칠 때만 작동한다. Node.js 상주 프로세스(또는 OS 스케줄러+CLI)가 매 틱마다 15명의 행동 결정을 순회하고 상태를 갱신해야 한다. 이것 없이는 국가가 아니라 턴제 보드게임이다.

### 2. 상태 내구성 (State Durability) — JSON은 프로덕션 저장소가 아님
registry.json과 economy.json이 SOT인데, 백업/복구/트랜잭션 보장이 전혀 없다. 틱 데몬이 economy.json 쓰기 중 프로세스가 죽으면 데이터가 반쪽짜리로 남는다. 최소한:
- (a) 틱마다 스냅샷 파일 회전 (state-YYYYMMDD-HHMM.json)
- (b) 쓰기 시 임시파일+rename 원자적 교체
- (c) 일일 백업 + 복구 CLI (`/persona restore 2026-04-05`)
SQLite 전환이 이상적이나 JSON 유지 시에도 이 세 가지는 필수.

### 3. 이벤트 버스 + 알림 (Event Bus + Notification)
결혼, 죽음, 위기, 돌파 같은 중요 이벤트가 발생해도 창조자가 알 방법이 없다. logs/*.jsonl에 기록은 되지만 **push 경로가 없다.** 틱 데몬 내에 이벤트 버스를 두고, 중요도 등급(긴급/중요/일반)별로 대시보드 토스트 또는 외부 웹훅(Discord/Slack/Telegram)으로 알림. 대시보드에 실시간 이벤트 피드 패널도 필요.

### 4. 멀티세션 격리 (Background Isolation)
페르소나 국가 시뮬레이션과 다른 작업(vibe-web, 포스리드, Trading Value)은 같은 Claude 세션 컨텍스트를 경쟁한다. 틱 데몬이 독립 프로세스로 돌면서 JSON을 갱신하고, Claude 세션은 필요할 때만 `/persona` 명령으로 개입하는 구조여야 한다. 시뮬레이션 코어 = LLM 호출 없이 돌아가는 순수 로직 레이어(PersonaBrain 추론 포함), LLM = 평가/대화/칙령 같은 고수준 개입에만 사용.

## 우선순위
틱 데몬(없으면 국가가 안 돌아감) > 상태 내구성(돌아가다 깨지면 복구 불가) > 이벤트 버스(창조자가 모르면 의미 없음) > 멀티세션 격리(다른 작업과 공존)
