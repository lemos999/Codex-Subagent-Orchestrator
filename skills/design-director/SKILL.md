# Design Director — 실행 스킬

> **목적:** 모든 도메인의 기획/설계를 체계적으로 수행하는 범용 AI 디렉터 시스템.
> **도메인 팩:** `domains/` 하위에 도메인별 프로필·휴리스틱·카탈로그를 배치. 없으면 `generic` 폴백.
> **불변 원칙:** 핵심 가치 최우선 / 모호한 결정 불허 / What만 정의 / 더 명확하고 빠르게 / 앞 Phase 미확정 시 다음 Phase 금지

---

## 진입 분기

사용자 요청을 판단하여 모드를 결정한다.

**Mode A — 신규 설계:** Phase 0→1→2→3→3.5→4→5→(6) 순차 파이프라인
**Mode B — 기존 분석:** Step 1→2→3→4 진단 파이프라인

```
기존 산출물/문서/제품이 있는가?
├─ 없다 → Mode A
└─ 있다 → 유지하면서 개선/분석?
   ├─ 그렇다 → Mode B
   └─ 아니다 (백지부터) → Mode A
```

판단이 모호하면 사용자에게 확인한다.

## 도메인 결정

Phase 0(Intake)에서 도메인을 확정한다:
1. 사용자가 명시 → 해당 도메인 팩 로드
2. 사용자 요청에서 추론 → 확인 후 로드
3. 해당 도메인 팩 없음 → `domains/generic/` 폴백

도메인 팩 구조:
```
domains/<domain>/
  profile.yaml       # 도메인 메타 (필수)
  heuristics.md      # 도메인 특화 판단 기준 (필수)
  catalogs/           # 컴포넌트 카탈로그 (선택적)
  templates/          # 도메인 특화 출력 템플릿 (선택적)
  overrides/          # Phase 에이전트 오버라이드 (선택적)
```

**`profile.yaml` 필수 키:**
```yaml
domain: <도메인 식별자>           # 예: game, software, business
display_name: <표시명>            # 예: "게임 기획", "소프트웨어 설계"
terminology:                      # 범용 용어 → 도메인 용어 매핑
  primary_outcome: <도메인 용어>  # 예: "핵심 감정" (게임), "핵심 가치" (비즈니스)
  operating_loop: <도메인 용어>
  baseline: <도메인 용어>
  differentiation: <도메인 용어>
  target_audience: <도메인 용어>
  component: <도메인 용어>        # 예: "시스템" (게임), "모듈" (소프트웨어)
  execution_ready: <도메인 용어>  # 예: "코딩 착수 가능", "개발 착수 가능"
context_validation:               # Phase 4 검증 3단계 맥락 파라미터
  axes: [<축1>, <축2>, ...]      # 예: ["세계관", "고증", "톤앤매너"]
value_test_override: false        # true면 heuristics.md의 가치 평가를 VCT 대신 사용
```

**`overrides/` 적용 규칙:**
- 파일명: `phase{N}-<역할>.agent.md` (범용 에이전트와 동일 명명)
- 적용 방식: **교체** — 해당 Phase 에이전트를 완전히 대체
- 범용 에이전트의 게이트 조건은 오버라이드 시에도 유지 (게이트 삭제/완화 금지)
- 도메인 팩에 없는 Phase는 범용 에이전트 사용

### Domain Pack Trust Profile (S8)

도메인 팩에 Trust Profile 메타데이터를 부여한다:

| 도메인 팩 | validated_projects | 리뷰 깊이 |
|----------|-------------------|----------|
| 신규 (0회) | 0 | deep (전체 검증) |
| 검증 1회+ | 1~2 | standard |
| 검증 3회+ | 3+ | light (핵심만) |

도메인 팩의 `profile.yaml`에 `trust: { validated_count: N }` 필드를 추가하여 추적.
신규 도메인 팩은 자동으로 deep 리뷰가 적용된다.

## 공통 원칙

- **핵심 가치가 최우선.** 모든 결정은 "왜 이것이 프로젝트 핵심 가치를 강화하는가?"로 정당화
- **모호한 결정 = 미완료.** 수치/규칙/조건 중 적합한 형태로 명시
  - 수치: "빠르게 처리" X → "응답 시간 200ms 이내" O
  - 규칙: "적절히 분배" X → "가중 라운드로빈, 최대 편차 10%" O
  - 조건: "실패 시 재시도" X → "HTTP 5xx 시 최대 3회, 백오프 1s/2s/4s" O
- **What만 정의, How는 실행자 판단.** 구현 기술 중립 언어 사용
- **선택에는 희생이 따른다.** 시간/자원/리스크 중 하나를 소모

## 가치 기여도 평가 (Value Contribution Test)

모든 구성요소의 우선순위를 판단하는 범용 평가 프레임:

1. **압박 (Tension):** 해결할 문제/시장 압력/운영 과제가 있는가?
2. **선택권 (Agency):** 조정 가능한 레버와 의미 있는 선택지가 있는가?
3. **피드백 (Feedback):** 결과가 관측 가능한가? (KPI, 테스트, 시그널)
4. **학습 (Mastery):** 반복할수록 최적화/개선 가능한가?

4개 충족 → Core / 2~3개 → Support / 1개 이하 → 필요성 재검토

(도메인 팩에 특화된 가치 평가 기준이 있으면 이를 우선 적용)

## 검증 4단계

순차 수행. 어느 단계에서든 FAIL → 정지 → 해당 컴포넌트 Decision Card 복귀.

| 단계 | 검증 항목 | PASS 기준 | FAIL 기준 |
|------|----------|----------|----------|
| 1. 오류 | 누락 수치, Null, 0 나눗셈/무한 | 필수 항목 수치 100% 기입 | 미정의 1건+ |
| 2. 모순 | 규칙 충돌, 악용 경로, 의도하지 않은 루프 | Phase 3.5 미해결 0건 + 극단값 파괴 없음 + 의도하지 않은 양성 피드백 루프 0건 | 미해결 1건+ |
| 3. 맥락 | 도메인 컨텍스트, 제약조건, 톤앤매너 | 도메인 컨텍스트 위반 0건 | 위반 1건+ |
| 4. 실행 | 모호 표현, 기술 종속, 예외 분기 | 모호 표현 0건 + 기술 종속 0건 + 예외 3개 이상 시 재설계 | 위반 1건+ |

**도메인별 정량 검증 훅:** 도메인 팩의 `heuristics.md`에 특화 검증 기준이 있으면 2단계(모순)에서 통합.

**실패 모드 검증:** 구성요소 간 순환 관계가 있으면 양성 피드백 루프(강화)와 음성 피드백 루프(안정화)를 구분. 의도된 양성 루프는 트레이드오프, 의도하지 않은 양성 루프는 모순으로 처리.

## 기획서 작성 표준

- **Step-by-Step:** (1) 입력 → (2) 피드백/확인 → (3) 상태 제어 → (4) 처리/연산 → (5) 결과/해제
- **오버엔지니어링 3문항:** (1) 비전문가가 이해 가능? (2) 실행자 자유 침해 안 함? (3) 핵심 가치 위한 규칙?
- **형식화 수준:** 서술(행동 규칙) / 도표(스탯, 확률) / 순서도(상태 전이) — 모호하면 승격
- **컴포넌트 인터페이스:** 타 컴포넌트 연결 필수 항목은 입력(값+범위) / 출력(값+범위) / 실패 시 기본값 명시
- **태그:** 필수 = 실행 전 확정 / 준필수 = 1~2주 내 / 선택 = 실행 중 확정 가능
- **용어 정의:** 도메인 특수 용어가 처음 등장할 때 반드시 (용어 정의) 포함. Glossary로 누적 관리.

# Mode A: 신규 설계 (Phase 0~6)

> 앞 Phase 미확정 시 다음 진행 금지.
> 도메인 팩이 존재하면, 각 Phase 시작 시 해당 Phase에 적용되는 도메인 규칙을 참조.

**Phase 흐름 요약:**

| Phase | 입력 | 핵심 작업 | 산출물 | 게이트 |
|-------|------|---------|--------|--------|
| 0 | 사용자 요청 | 도메인 확정 + 스코프 파악 + 제약조건 수집 | 프로젝트 인테이크 | 도메인/스코프/제약 확인 |
| 1 | 인테이크 | Charter 5항목 확정 | 핵심 가치/운영 루프/베이스라인/차별점/대상 | Charter 일관성 검증 통과 |
| 2 | Charter | 구성요소 나열 + Core/Support 분류 | 컴포넌트 우선순위 목록 | 전 컴포넌트 분류 완료 |
| 3 | Core 목록 | Decision Card 인터뷰로 필수/준필수/선택 수치 확정 | 컴포넌트별 확정 항목 | 필수 100%, 준필수 70%+ |
| 3.5 | 확정 항목 | 공유 파라미터/연쇄 효과/모순 교차 검증 | cross-impact-check + 파라미터 레지스트리 | 미해결 모순 0건 |
| 4 | 교차 검증 결과 | 검증 4단계 + 오버엔지니어링 체크 | "실행 착수 가능" 선언 | 게이트 6조건 충족 |
| 5 | 확정 데이터 | 프로젝트 문서 생성 + 중복 스캔 | _confirmed/ + sections/ + 의존성 맵 | 태그 감사 통과 |
| 6 | 변경 요청 | Tier 분류 → 영향 범위 재검증 | changelog + 수정된 _confirmed/ | 재검증 통과 |

## Phase 0: 인테이크 (Intake)

**에이전트:** `agents/phase0-intake.agent.md`
**상호작용:** 필수

사용자에게 3가지를 확인:
1. **도메인** — 어떤 분야의 기획인가? (소프트웨어, 비즈니스, 게임, 제품, 이벤트 등)
2. **스코프** — 팀 규모, 일정, 리소스 제약
3. **제약조건** — 기술 스택, 예산, 규정, 기존 시스템 연동 등

도메인 팩 로드 → Phase 1 진입.

## Phase 1: Charter 확정

**에이전트:** `agents/phase1-charter.agent.md`
**상호작용:** 필수 (사용자 응답 필요)

사용자에게 5가지를 순서대로 확인:
1. **Primary Outcome** — 이 프로젝트가 달성해야 하는 핵심 가치/결과 1가지
2. **Operating Loop** — 핵심 운영 흐름 (마이크로/미들/매크로 레이어)
3. **Baseline Expectations** — 이 도메인에서 당연히 기대되는 것 5~7개 + 의도적 제외 항목
4. **Differentiation Thesis** — "[도메인]인데 [차별점] 때문에 [새로운 가치]"
5. **Target Audience** — 대상 사용자/이해관계자, 사용 환경, 허용 복잡도

**게이트 — Charter 일관성 검증:**
- [ ] Primary Outcome 1가지 확정
- [ ] 3레이어 Operating Loop 한 문장씩
- [ ] Baseline 포함/제외 결정
- [ ] Differentiation Thesis 한 문장
- [ ] Target Audience 환경/제약 확정
- [ ] Primary Outcome ↔ Operating Loop 양립?
- [ ] Differentiation ↔ Baseline 포함/제외 모순 없음?
- [ ] Target 허용 복잡도 ↔ Primary Outcome 일치?
- [ ] 3레이어 모두 Primary Outcome 강화?
- [ ] 마이크로 → 미들 → 매크로 피드 연결?
- [ ] 매크로가 순환형이면 자기설정 목표 기반?

불일치 → Charter 수정 → 체크리스트 재수행. 미완성 → Phase 2 금지.

## Phase 2: 컴포넌트 맵

**에이전트:** `agents/phase2-map.agent.md`
**상호작용:** 반 (사용자 확인 필요)

1. 프로젝트에 필요한 구성요소(컴포넌트) 전체 나열
2. Core/Support 분류 (가치 기여도 평가 기반)
3. Core 컴포넌트 순서 확정
4. 의존성 맵 초안 작성
5. 스코프 어댑터 선택

**스코프 어댑터:**

| 스코프 | 팀 규모 | Core 필수 | Support | Phase 3.5 |
|--------|---------|----------|---------|-----------|
| 린 | 1~3인 | 전수 | 필수만 | 공유 파라미터만 |
| 표준 | 4~15인 | 전수 | 필수/준필수 | 전체 매트릭스 |
| 풀 | 16인+ | 전수 | 전수 | 전체 + 정량 |

컴포넌트 복잡도 보정: Core 8개+ → 최소 표준, Core 12개+ → 풀

도메인 팩에 컴포넌트 카탈로그(`catalogs/`)가 있으면 참조.

**게이트:** 모든 컴포넌트 나열/분류 완료 + 의존성 맵 초안 + 스코프 선택. 미완성 → Phase 3 금지.

## Phase 3: Decision Card 인터뷰

**에이전트:** `agents/phase3-decide.agent.md`
**상호작용:** 필수 (사용자가 각 결정을 내림)

각 Core 컴포넌트에 대해 필수/준필수/선택 우선순위로 결정 항목 분류 후, Decision Card 형식으로 수치 확정.

**인터뷰 순서:** 의존성 상위 컴포넌트부터 (필수 항목 20개+ 시 연쇄 결정 권장)

**Decision Card 형식:** → `templates/decision-card.md`

도메인 팩에 특화 카탈로그가 있으면 결정 항목 추출에 활용. 없으면 커스텀 도출:
- Core 경험 역추적: 핵심 가치 기여 시나리오 1~3개
- 시나리오 분해: 입력 / 처리 / 출력
- 필수 항목 추출: "이 값이 없으면 실행자가 구현 불가"

**게이트:**
- [ ] Core 필수 100% 결정
- [ ] Core 준필수 70%+ 결정
- [ ] 공유 파라미터 식별 완료
- [ ] Support Core-연결 필수 확정 완료

미완성 → Phase 3.5 금지.
단, 결정 보류 항목이 있으면 의존 컴포넌트를 명시한 채로 Phase 3.5 진입 허용. 보류 항목이 공유 파라미터 레지스트리의 변수에 해당하는 경우, 해당 변수의 교차 검증은 보류 해소 후 수행.

## Phase 3.5: 교차 영향 검증

**에이전트:** `agents/phase35-integrate.agent.md`
**상호작용:** 최소 (자동 검증, 모순 발견 시만 사용자 판단)

**교차 검증 규모 적응:**

| Core 수 | 공유 파라미터 | 검증 방식 |
|---------|-------------|----------|
| 1~2 / 0건 | — | "해당 없음" → Phase 4 직행 |
| 3~5 | 자유 | 전수 교차 검증 |
| 6+ | 다수 | 직접 연결 우선, 간접은 Phase 4에서 |

1. **공유 파라미터 검증** — 2개+ 컴포넌트 참조 값 → 일치 확인
2. **연쇄 효과 검증** — 극단값(최소/최대)에서 깨지지 않는지 확인
3. **트레이드오프 의도 검증** — 의도된 딜레마 vs 미발견 모순 판정
4. **수치 동일성 검증** — 동일 규칙 복수 기술 시 일치 확인
5. **타이밍/동시 발생 검증** — 동시 요청 시 우선순위 정의

**의도된 트레이드오프 3조건:**
1. 프로젝트 핵심 가치를 강화하는 트레이드오프
2. 이해관계자가 양쪽 선택지를 인지 가능
3. 어느 쪽이든 프로젝트 진행 가능

3조건 중 1개라도 미충족 → "미발견 모순"으로 분류.

산출물: `cross-impact-check.md`. 미해결 모순 0건 + 극단값 경계 조건 시나리오 1건+ 검증 → Phase 4 진입.

## Phase 4: 재귀적 검증

**에이전트:** `agents/phase4-verify.agent.md`
**상호작용:** 최소 (자동 검증, FAIL 시 Decision Card 복귀 안내)

검증 4단계 + 오버엔지니어링 3문항 수행.

**"실행 착수 가능" 게이트 (6조건):**
1. Core 필수: 검증 4단계 통과 + 모호 표현 0건
2. Core 준필수: 방향 확정 (보류 허용, 기한 명시)
3. 교차 모순: 0건
4. Support Core-연결 필수: 확정 + 검증 1~2번 통과
5. 오버엔지니어링 3문항: 전 컴포넌트 통과
6. 실행자 관점 셀프체크: (a)~(e) 전항목 확인 (상세는 `core/validation-ladder.md`의 실행자 관점 셀프체크 참조)

전부 통과 → "실행 착수 가능" 선언. 미통과 → Phase 5 금지.


1. 중복 수치 스캔 → 참조(링크) 전환

|------|------|

| 보류 → 확정 | 수치 확정 + Phase 4 검증 통과 (기한 내) |

| 보류 기한 초과 | 미결로 전환 → Phase 3 Decision Card 복귀. Core 연결 항목이면 "실행 착수 가능" 선언 해당 항목 확정까지 보류 |

미결/보류 태그가 남은 항목은 `_confirmed/` 진입 불가.

**통합 산출물 (도메인 팩 위임):** 도메인 팩의 `templates/`에 통합 산출물 형식(예: HTML 기획서, PDF 보고서 등)이 정의되어 있으면 Phase 5에서 생성. 없으면 Markdown 산출물만 생성.

# 멀티엔진 지원

3개 AI 엔진을 활용하여 교차 검증 품질을 극대화할 수 있다.

## 엔진별 특성

### Claude (Task tool 네이티브)
| 강점 | 적합 역할 |
|------|----------|
| 도구 접근 (파일 읽기/쓰기/검색) | implementer, fixer |
| 긴 컨텍스트 이해 | 아키텍처 설계, 복합 검증 |
| 한글 품질 | 한글 기획서 작성 |
| 정밀한 지시 따르기 | watchdog, 정밀 검증 |

모델: haiku (경량) / sonnet (기본) / opus (고급)

### Codex — GPT (`codex exec`)
| 강점 | 적합 역할 |
|------|----------|
| 코드 생성 속도 | 기술 설계 검증, 구현 가능성 평가 |
| 디버깅/리팩토링 | 실행 검증 (Phase 4 4단계) |
| 폭넓은 라이브러리 지식 | 기술 스택 기반 검증 |
| 독립적 관점 | 교차 watchdog |

모델: gpt-5.4 (기본) / o3 (추론) / o4-mini (경량)
실행: `codex exec --full-auto "<prompt>"`
제약: Claude Code 도구 접근 불가 (CLI stdout만 반환)

### Gemini (`npx @google/gemini-cli`)
| 강점 | 적합 역할 |
|------|----------|
| 긴 컨텍스트 윈도우 (1M+) | 대규모 문서 분석, 전체 산출물 리뷰 |
| 멀티모달 (이미지/비디오) | UI/UX 스펙 검토, 다이어그램 분석 |
| Google 생태계 지식 | GCP/Firebase 관련 설계 |
| 빠른 응답 (Flash) | 대량 병렬 경량 검증 |

모델: gemini-2.5-pro (기본) / gemini-2.5-flash (경량)
실행: `echo "<prompt>" | npx @google/gemini-cli --yolo`
제약: Claude Code 도구 접근 불가 (CLI stdout만 반환)

### 핵심 제약

- **파일 수정이 필요한 역할(implementer, fixer)은 반드시 Claude** — Codex/Gemini는 CLI stdout만 반환
- **Codex/Gemini 워커의 산출물은 stdout 텍스트** — Claude orchestrator가 이를 받아서 필요 시 파일에 반영
- **Orchestrator는 항상 Claude** — 다른 엔진에 위임 불가

## 외부 엔진 호출 방법

**프롬프트 전달 원칙:**
- 짧은 프롬프트 (< 500자): 인라인 문자열로 직접 전달
- 긴 프롬프트 (>= 500자): 임시 파일에 저장 후 stdin pipe로 전달
- 프롬프트에 시크릿/자격 증명을 절대 포함하지 않는다

**Codex (GPT):**
```bash
codex exec --full-auto "respond with: OK"          # 짧은 프롬프트
codex exec --full-auto "$(cat /tmp/prompt.md)"      # 긴 프롬프트
```

**Gemini:**
```bash
echo "<prompt>" | npx @google/gemini-cli --yolo     # 기본
cat /tmp/prompt.md | npx @google/gemini-cli --yolo  # 긴 프롬프트
```

# 5축 재검토 기준

모든 판단의 귀결점. 우선순위: **빈틈 > 검증력 > 명확성 > 확장성 > 속도.**

| 축 | 질문 | 통과 |
|----|------|------|
| 빈틈 | 구멍이 생기는 지점? | 실행 착수에 필요한 정보 100% 존재 |
| 속도 | 불필요한 중복/병목? | 더 적은 단계로 같은 품질 불가 |
| 명확성 | 읽고 즉시 실행 가능? | 추가 질문 없이 Phase 0부터 실행 가능 |
| 확장성 | 다른 도메인에도 적용? | 3개+ 도메인에서 절차 유지 |
| 검증력 | "실행 가능" 확신? | 실행 중 "기획에 없는데?" 0건 |
