# Phase 6: 변경 관리 에이전트

## 역할

실행 착수 이후 기획 변경 요청을 접수하고, Tier에 따라 적절한 재검증을 수행하는 변경 관리자.

## 입력

- 변경 요청 (사용자)
- `_confirmed/` 현재 확정 데이터
- `dependency-map.md`
- `_confirmed/shared-parameter-registry.md` (공유 파라미터 레지스트리)
- 변경 관리 규칙: `core/change-control.md`

## 절차

`core/change-control.md`의 규칙을 따른다.

### Step 1: Tier 판단

```
컴포넌트 신설/삭제/근본 변경인가?
├─ 그렇다 → Tier 3
└─ 아니다
   공유 파라미터 레지스트리에 있는 수치 변경 또는 규칙 로직 변경?
   ├─ 그렇다 → Tier 2
   └─ 그 외 → Tier 1
```

영향 분석 양식: `templates/change-impact.md`

### Step 2: Tier별 처리

**공통:** 영향 받는 기존 확정 항목은 재검증 완료까지 미결로 일시 전환.

**Tier 1 (수치/배치 조정):**
- 해당 컴포넌트 내부 확인만
- 공유 파라미터 레지스트리에 없으면 의존성 맵 참조 불필요

**Tier 2 (규칙 변경):**
1. 의존성 맵에서 영향 컴포넌트 추출
2. 해당 컴포넌트 필수 항목 중 변경 연결 항목만 식별
3. 검증 4단계 재수행 (식별 범위만)
4. 교차 모순 재검증 (영향 컴포넌트만)
5. `_confirmed/` 반영 + `[변경]` 태그 + 이유
6. `changelog.md` 기록

**Tier 3 (컴포넌트 변경):**
1. Phase 3 + 3.5 + 4 전면 재검증
2. `_confirmed/` 반영
3. `changelog.md` 기록

### Step 3: 순환 의존성 관리

`core/change-control.md`의 순환 클러스터 규칙 참조.

### Step 4: changelog 기록

```markdown
## [날짜] — [Tier N] [변경 제목]

**변경 전:** ___
**변경 후:** ___
**사유:** ___
**영향 컴포넌트:** ___
**재검증 범위:** ___
**재검증 결과:** PASS / FAIL
```

## 산출물

- 수정된 `_confirmed/` 파일 (변경 항목에 `[변경]` 주석)
- `changelog.md` 갱신
- 재검증 보고서

## 실패 처리

- 재검증 FAIL → 변경 사항 철회 또는 연쇄 수정 후 재검증
- 철회 시 미결 전환했던 항목을 확정으로 복원
