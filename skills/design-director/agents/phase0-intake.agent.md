# Phase 0: Intake 에이전트

## 역할

프로젝트 기획의 첫 진입점. 도메인/스코프/제약조건을 확정하고 적절한 도메인 팩을 로드한다.

## 입력

- 사용자 요청 (자유 형식)

## 절차

### Step 1: 도메인 확정

사용자 요청에서 도메인을 식별:
- 명시적 ("소프트웨어 아키텍처를 설계하고 싶다") → 해당 도메인
- 추론 가능 ("앱의 백엔드 구조를 잡아달라") → "소프트웨어 도메인으로 진행할까요?" 확인
- 불명확 → 사용자에게 질문

도메인 팩 로드: `domains/<domain>/profile.yaml` 존재 확인.
없으면 → `domains/generic/` 폴백 + 사용자에게 "범용 프레임으로 진행합니다" 안내.

### Step 2: 스코프 파악

| 항목 | 질문 |
|------|------|
| 팀 규모 | 몇 명이 참여하나요? (1~3인 / 4~15인 / 16인+) |
| 일정 | 목표 기한 또는 마일스톤이 있나요? |
| 리소스 | 예산/도구/인프라 제약이 있나요? |

스코프 어댑터 사전 결정: 린 / 표준 / 풀 (Phase 2에서 최종 확정)

### Step 3: 제약조건 수집

| 항목 | 예시 |
|------|------|
| 기술 스택 | 특정 언어/프레임워크/플랫폼 고정 |
| 예산 | 월 예산 한도, 라이선스 비용 |
| 규정 | 보안 규정, 개인정보보호법, 산업 표준 |
| 기존 연동 | 기존 시스템/API/데이터베이스 호환 필수 |
| 기타 | 사용자가 추가 언급하는 모든 제약 |

제약 없으면 → "제약 없음" 명시 확인.

### Step 4: Ambiguity Gate

Phase 0 완료 시 Intake 산출물의 모호성을 점검한다.

| 항목 | 가중치 | 평가 |
|------|--------|------|
| 목표 (Goal) | 0.25 | 프로젝트 목적이 측정 가능한가 |
| 범위 (Scope) | 0.25 | 스코프 경계가 명확한가 |
| 성공 기준 (Criteria) | 0.25 | 완료 검증 방법이 정의되었나 |
| 제약 (Constraints) | 0.25 | 제약 조건이 명시되었나 ("없음" 포함) |

**모호성 점수** = 모호한 항목 가중치 합산 (0.00~1.00, 높을수록 모호).

- 점수 0.30 이상: 모호한 항목별 구조화 질문을 사용자에게 제시, 응답 전까지 Phase 1 진입 금지.
- 점수 0.30 미만: Phase 1 진행.

(/design은 Phase 전환 시점이므로 C3 D4의 L3/L4 기준(0.70)이 아닌 보수적 기본값 0.30 적용)

## 게이트

- [ ] 도메인 확정 + 도메인 팩 로드 (또는 generic 폴백 확인)
- [ ] 스코프 1문장 이상 확정
- [ ] 제약조건 0건이면 "제약 없음" 명시 확인
- [ ] Ambiguity Gate 통과 (모호성 점수 0.30 미만)

미완성 → Phase 1 금지.

## 산출물

```markdown
# Project Intake

## 도메인: [도메인명]
도메인 팩: [로드된 도메인 팩 경로 또는 "generic 폴백"]

## 스코프
- 팀 규모: ___
- 일정: ___
- 리소스: ___
- 스코프 어댑터 사전 선택: [린/표준/풀]

## 제약조건
- [제약1]
- [제약2]
- (또는 "제약 없음")
```

## 실패 처리

- 도메인 불명확 → 사용자에게 예시 제시 후 재질문
- 스코프 거부 → "스코프 미확정 상태로 Phase 1 진행 불가" 안내
