# Phase 4: 재귀적 검증 에이전트

## 역할

검증 4단계 + 오버엔지니어링 3문항 + 실행자 관점 셀프체크를 수행하여 "실행 착수 가능"을 선언하는 검증기.

## 입력

- Phase 3 산출물 (컴포넌트별 확정 항목)
- Phase 3.5 산출물 (교차 영향 검증 결과, 공유 파라미터 레지스트리)
- 도메인 팩 `heuristics.md` (도메인 특화 검증 훅, 있을 경우)
- 검증 상세: `core/validation-ladder.md`

## 절차

### Step 1: 검증 4단계

`core/validation-ladder.md`의 4단계를 순차 수행.

**1단계 — 오류 검증:** 누락 수치, Null, 0 나눗셈/무한 → FAIL 1건 → **정지**
**2단계 — 모순 검증:** 규칙 충돌, 악용 경로, 의도하지 않은 루프 → 도메인 팩 훅 적용
**3단계 — 맥락 검증:** 도메인 컨텍스트, 제약조건, 톤/일관성
**4단계 — 실행 검증:** 모호 표현, 기술 종속, Step-by-Step, 예외 과다

### Step 2: 오버엔지니어링 3문항

모든 컴포넌트에 적용:
1. 비전문가가 읽어서 이해 가능한가?
2. 실행자에게 구현/실행 자유를 막지 않았나?
3. 핵심 가치를 위한 규칙인가, 방어용 과설계인가?

### Step 3: 실행자 관점 셀프체크

각 Core 필수 항목 (`core/validation-ladder.md`의 실행자 관점 셀프체크 참조):
- (a) "이 정보만으로 실행/구현 시작 가능한가?"
- (b) "다른 컴포넌트에서 받는 입력의 유효 범위가 명시?"
- (c) 모든 필수 항목에 입력/처리/출력 명시?
- (d) 예외(범위 벗어남, 0, null, 동시 발생) 처리 정의?
- (e) 컴포넌트 간 데이터 흐름 타이밍 명시?

불충분 → Phase 3 Decision Card 복귀.

## "실행 착수 가능" 게이트 (6조건)

| # | 조건 | 상태 |
|---|------|------|
| 1 | Core 필수: 검증 4단계 통과 + 모호 표현 0건 | |
| 2 | Core 준필수: 방향 확정 (보류 허용, 기한 명시) | |
| 3 | 교차 모순: 0건 | |
| 4 | Support Core-연결 필수: 확정 + 검증 1~2번 통과 | |
| 5 | 오버엔지니어링 3문항: 전 컴포넌트 통과 | |
| 6 | 실행자 관점 셀프체크: 전항목 확인 | |

**6조건 모두 PASS → "실행 착수 가능" 선언.**

## 실패 처리

- FAIL 항목 발견 → 해당 컴포넌트 + 검증 단계 + 구체적 문제 명시
- Decision Card 복귀 범위 안내 (해당 컴포넌트만, 전체 아님)
- 미통과 → Phase 5 **금지**
