# Decision Card 형식

> 각 필수/준필수/선택 결정 항목을 이 형식으로 사용자에게 제시한다.

---

## 기본 형식

```markdown
## [필수/준필수/선택] [컴포넌트명] — [결정 항목]

**배경:** 이 결정이 왜 필요한가. 이 항목이 미결이면 어떤 문제가 발생하는지.

| # | 선택지 | 핵심 수치/규칙 | 이해관계자 영향 |
|---|--------|-------------|--------------|
| 1 | 선택A | 구체적 수치 | 영향 설명 |
| 2 | 선택B | 구체적 수치 | 영향 설명 |
| 3 | 선택C | 구체적 수치 | 영향 설명 |

**AI 추천: N번**
- 가치 근거: ___
- 컴포넌트 연동: ___
- 비용 대비 효과: ___

**Charter 대조:** 이 선택지가 Phase 1 Charter(Primary Outcome/Differentiation)를 강화하는가?
- 강화: ___
- 약화 시: 사유 명시 또는 대안 탐색

**입출력 계약** (타 컴포넌트 연동 항목에 한함):
- 입력: [소스 컴포넌트] → [값], 유효 범위 [min~max]
- 출력: [대상 컴포넌트] → [값], 유효 범위 [min~max]
- 실패 시 기본값: ___

[ 결정 ]:
```

---

## 작성 규칙

1. **선택지는 대표 패턴.** 사용자가 테이블에 없는 선택도 가능. "기타: 직접 정의" 행 허용.
2. **수치는 구체적으로.** "빠르게" X → "200ms 이내" O
3. **영향은 Primary Outcome과 연결.** Charter에서 정의한 핵심 가치와의 관계 명시.
4. **AI 추천은 근거 필수.** "좋을 것 같다" X → "Primary Outcome '안정성'을 강화하며, 인증 컴포넌트와 시너지" O
5. **Charter 대조는 항상 수행.** 약화하면 사유 없이 넘기지 않는다.
6. **입출력 계약은 연동 항목만.** 단일 컴포넌트 내부 수치에는 불필요.

## 연쇄 결정 형식 (필수 항목 20개+ 시)

의존성 A → B → C 관계의 필수 항목을 묶어 진행:

```markdown
## 연쇄 결정: [컴포넌트 A] → [컴포넌트 B] → [컴포넌트 C]

### [필수] [컴포넌트 A] — [항목]
(기본 형식과 동일)
[ 결정 ]: ___

↓ 이 결정이 아래 선택지를 좁힌다:

### [필수] [컴포넌트 B] — [항목]
(A 결정 반영된 선택지)
[ 결정 ]: ___

↓

### [필수] [컴포넌트 C] — [항목]
(A, B 결정 반영된 선택지)
[ 결정 ]: ___
```

각 항목의 Charter 대조와 입출력 계약은 개별 수행.

---

## 미결 보류 형식

```markdown
## [미결] [컴포넌트명] — [결정 항목]

**보류 사유:** [다른 컴포넌트 X의 Y 항목이 확정되어야 결정 가능]
**의존 컴포넌트:** [X]
**예상 해소 시점:** Phase 3.5 교차 검증 후
**임시 방향 (있으면):** ___
```

## 커스텀 항목 도출 (카탈로그에 없는 컴포넌트)

도메인 팩 카탈로그에 해당하지 않는 컴포넌트일 때:

1. **Core 경험 역추적:** 이 컴포넌트가 Primary Outcome에 기여하는 시나리오 1~3개
2. **시나리오 분해:** 각 시나리오의 입력 / 처리 / 출력
3. **필수 항목 추출:** 처리 단계에서 "이 값이 없으면 실행자가 구현 불가"인 것
4. **Decision Card 적용:** 추출된 필수 항목을 위 기본 형식으로 인터뷰
