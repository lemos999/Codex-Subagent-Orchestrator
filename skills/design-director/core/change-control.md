# Change Control — 변경 관리 규칙

> 실행 착수 이후 기획 변경 요청을 체계적으로 처리.

---

## Tier 판단 흐름

```
컴포넌트 신설/삭제/근본 변경인가?
├─ 그렇다 → Tier 3
└─ 아니다
   공유 파라미터 변경 또는 규칙 로직 변경?
   ├─ 그렇다 → Tier 2
   └─ 그 외 → Tier 1
```

---

## Tier별 처리

### Tier 1 (수치/배치 조정)

- 해당 컴포넌트 내부 확인만
- 공유 파라미터 레지스트리에 없으면 의존성 맵 참조 불필요 (빠른 경로)
- 영향 받는 확정 항목: 재검증 완료까지 미결 전환

### Tier 2 (규칙 변경)

1. 의존성 맵에서 영향 컴포넌트 추출
2. 해당 컴포넌트 필수 항목 중 변경 연결 항목만 식별
3. 검증 4단계 재수행 (Phase 4, 식별 범위만)
4. 교차 모순 재검증 (Phase 3.5, 영향 컴포넌트만)
5. `_confirmed/` 반영 + `[변경]` 태그 + 이유
6. `changelog.md` 기록
7. 영향 받는 확정 항목: 재검증 완료까지 미결 전환

### Tier 3 (컴포넌트 변경)

1. Phase 3 + 3.5 + 4 전면 재검증
2. `_confirmed/` 반영
3. `changelog.md` 기록
4. 영향 받는 확정 항목: 재검증 완료까지 미결 전환

---

## 순환 의존성 관리

의존성 맵에 A->B->C->A 순환이 있으면:
1. 순환 참여 컴포넌트를 "순환 클러스터"로 식별
2. 클러스터 내 Tier 2 -> 클러스터 범위만 재검증
3. 클러스터 간 -> 영향받는 클러스터만

---

## Changelog 형식

```markdown
## [날짜] — [Tier N] [변경 제목]

**변경 전:** ___
**변경 후:** ___
**사유:** ___
**영향 컴포넌트:** ___
**재검증 범위:** ___
**재검증 결과:** PASS / FAIL
```

---

## 실패 처리

- 재검증 FAIL -> 변경 사항 철회 또는 연쇄 수정 후 재검증
- 철회 시 미결 전환했던 항목을 확정으로 복원
