# Validation Ladder — 검증 4단계

> 모든 확정 항목에 순차 적용. 어느 단계에서든 FAIL -> 정지 -> 해당 컴포넌트 Decision Card 복귀.

---

## 1단계: 오류 검증 (Error)

| 체크 | PASS 기준 |
|------|----------|
| 누락 수치 | 모든 필수 항목의 수치 필드 100% 기입 |
| Null 참조 | "미정", "TBD", "나중에" 0건 |
| 0 나눗셈/무한 | 분모가 0이 되는 경로 없음, 무한 증가 경로 없음 |

FAIL 1건 -> **정지** -> Phase 3 해당 컴포넌트 복귀

---

## 2단계: 모순 검증 (Contradiction)

| 체크 | PASS 기준 |
|------|----------|
| 규칙 충돌 | Phase 3.5 미해결 모순 0건 |
| 악용 경로 | 설계 의도를 우회하는 경로 0건 + 극단값 투입 시 연쇄 파괴 없음 |
| 의도하지 않은 루프 | 양성 피드백 루프(무한 성장/죽음의 나선) 0건 |

**악용 경로 판정:** "설계 의도를 우회하여 의도하지 않은 결과를 얻는 경로"
- 소프트웨어: 권한 우회, 무한 재시도, 레이스 컨디션
- 비즈니스: 규정 우회, 인센티브 악용, 자원 무한 소비
- 게임: 무한 자원, 무한 스턴, 의도하지 않은 해법

**피드백 루프 검증:** 구성요소 간 순환 관계에서:
- 양성 피드백 루프(강화) vs 음성 피드백 루프(안정화) 구분
- 의도된 양성 루프 = 트레이드오프 (허용)
- 의도하지 않은 양성 루프 = 모순 (수정 필요)

**도메인 정량 검증 훅:** 도메인 팩 `heuristics.md`에 특화 검증 기준이 있으면 이 단계에서 통합 적용.

---

## 3단계: 맥락 검증 (Context)

| 체크 | PASS 기준 |
|------|----------|
| 도메인 컨텍스트 | 도메인 팩 `profile.yaml`의 `context_validation.axes` 항목별 위반 0건 |
| 제약조건 | Phase 0에서 수집한 제약조건 위반 0건 |
| 톤/일관성 | 산출물 전체의 표현 수준/일관성 위반 0건 |

도메인 팩에 `context_validation.axes`가 정의되지 않았으면 기본 축: ["도메인 관행", "제약조건", "표현 일관성"]

---

## 4단계: 실행 검증 (Execution)

| 체크 | PASS 기준 |
|------|----------|
| 모호 표현 | "적절히", "필요에 따라" 같은 비정량 표현 0건 |
| 기술 종속 | 특정 프레임워크/엔진/언어 강요 표현 0건 |
| Step-by-Step | 모든 컴포넌트가 입력->처리->출력 형식 |
| 예외 과다 | 단일 규칙 예외 3개+ -> 규칙 재설계 |

---

## 오버엔지니어링 3문항

모든 컴포넌트에 적용:

1. **비전문가가 읽어서 이해 가능한가?**
2. **실행자에게 구현/실행 자유를 막지 않았나?**
3. **핵심 가치를 위한 규칙인가, 방어용 과설계인가?**

하나라도 NO -> 해당 규칙 간소화 또는 제거.

---

## 실행자 관점 셀프체크

각 Core 필수 항목:
- (a) "이 정보만으로 실행/구현 시작 가능한가?"
- (b) "다른 컴포넌트에서 받는 입력의 유효 범위가 명시?"
- (c) 모든 필수 항목에 입력/처리/출력 명시?
- (d) 예외(범위 벗어남, 0, null, 동시 발생) 처리 정의?
  - (i) 같은 파라미터에 2개+ 효과 적용 순서
  - (ii) 상태 2개+ 동시 부여 시 우선순위
  - (iii) 상태 전이 중 추가 트리거 발생 시 처리
- (e) 컴포넌트 간 데이터 흐름 타이밍(동기/비동기/이벤트 기반) 명시?

불충분 -> Phase 3 Decision Card 복귀.
