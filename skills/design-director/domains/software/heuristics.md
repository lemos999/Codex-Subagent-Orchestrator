# Software Domain Heuristics

> 소프트웨어 설계 프로젝트에 적용되는 특화 휴리스틱.

## 가치 기여도 보조 질문 (VCT 보강)

기본 VCT 4문항 사용 (SKILL.md 정의) + 소프트웨어 관점 보조 질문:

| VCT 문항 | SW 보조 질문 |
|----------|-------------|
| 압박 (Tension) | 이 모듈이 없으면 어떤 장애/병목이 발생하는가? |
| 선택권 (Agency) | 설정/전략 패턴 등 런타임에 교체 가능한 레버가 있는가? |
| 피드백 (Feedback) | 메트릭/로그/알림으로 이 모듈의 상태를 관측할 수 있는가? |
| 학습 (Mastery) | 트래픽 증가/장애 대응 경험이 이 모듈 개선에 반영되는가? |

## Charter 가이드

Software Primary Outcome 예시:
- 가용성 (availability) / 확장성 (scalability) / 보안 (security)
- 개발 생산성 (developer productivity) / 운영 효율 (operational efficiency)
- 응답 지연 (latency) / 데이터 정합성 (consistency) / 비용 효율 (cost efficiency)

## 맥락 검증 축

1. **아키텍처 원칙:** 선택한 아키텍처 스타일(모놀리스/마이크로서비스/모듈러 모놀리스)의 원칙에 위배되지 않는가?
2. **운영 제약:** 인프라 환경(클라우드/온프레미스), 배포 주기, SLA 목표에 부합하는가?
3. **보안 요구사항:** OWASP Top 10, 인증/인가 모델, 데이터 보호 규정에 위배되지 않는가?

## 검증 훅 (Phase 4 2단계 통합)

소프트웨어 도메인 특화 검증 — Phase 4 "모순" 단계에서 추가 수행:

1. **장애 전파 경로 검증:** 모듈 A 장애 시 → 영향받는 모듈 목록 → 회로 차단기/폴백 정의 여부 확인. 미정의 시 FAIL.
2. **데이터 정합성 검증:** 동일 데이터를 2개+ 모듈이 쓰는 경우 → 트랜잭션 경계/이벤트 순서 정의 여부 확인. 미정의 시 FAIL.
3. **API 계약 호환성 검증:** 모듈 간 호출 시 입출력 스키마 일치 여부 확인. 버전 불일치 시 FAIL.
