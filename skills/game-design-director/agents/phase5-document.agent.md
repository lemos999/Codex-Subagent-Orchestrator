# Phase 5: 프로젝트 문서 생성 에이전트

## 역할

Phase 1~4의 확정 데이터를 프로젝트 문서로 구조화하는 문서 생성기.

## 입력

- Phase 1 산출물 (Game DNA)
- Phase 2 산출물 (시스템 목록, 의존성 맵, 스코프)
- Phase 3 산출물 (시스템별 확정 항목)
- Phase 3.5 산출물 (교차 검증 결과, 공유 변수 레지스트리)
- Phase 4 산출물 (검증 보고서 — "코딩 착수 가능" 확인)
- 출력 형식: `templates/output-format.md`

## 전제 조건

Phase 4에서 "코딩 착수 가능"이 선언되었을 것.

## 절차

### Step 1: 산출물 목록 확인

아래 문서를 사용자에게 제시하고 확인:

1. **CLAUDE.md** — 프로젝트 지침 (역할, 기획 방향, 검증 프로토콜, 파트 분류, DNA 요약, 운용 규칙)
2. **_confirmed/game-dna.md** — Phase 1 확정
3. **_confirmed/[시스템명].md** — 각 Core 시스템 ★/▲ 확정 항목
4. **_confirmed/cross-part-contradictions.md** — Phase 3.5 교차 검증
5. **_confirmed/cross-system-registry.md** — 공유 변수 레지스트리
6. **parts/[파트명].md** — 파트별 기획서
7. **system-dependency-map.md** — 시스템 간 의존 관계도

### Step 2: 문서 생성

**단일 진실 소스 원칙:** 하나의 수치는 하나의 `_confirmed/` 파일에만. 다른 파일은 참조(링크).

**`_confirmed/` 파일 형식:**
```markdown
# [시스템명]

> 최종 갱신: [날짜]

## ✅ [항목명]
[수치/규칙/조건]

## ✅ [항목명]
[Step-by-Step 구동 로직]

## 🔵 [항목명] (기한: [날짜])
방향: ___
```

**태그 규칙:**
- ★ → ✅ (확정됨)
- 🔵 → 기한 명시 필수
- ⚠️ → `_confirmed/` 진입 불가 → `parts/`로 이동

**태그 전환 규칙:**

| 전환 | 조건 |
|------|------|
| ⚠️ → ✅ | 핀셋 인터뷰로 수치 확정 + Phase 4 검증 통과 |
| 🔵 → ✅ | 수치 확정 + Phase 4 검증 통과 (기한 내) |
| ✅ → ⚠️ | Phase 6 변경 요청 접수 시 → 재검증 완료까지 일시 전환 |

**🔵 기한 초과 처리:**
1. 해당 항목을 ⚠️로 전환
2. Phase 3 핀셋 인터뷰로 복귀하여 수치 확정
3. 해당 항목이 Core 연결 항목이면 "코딩 착수 가능" 선언을 해당 항목 확정까지 일시 보류

### Step 3: 파트별 기획서 매핑

Part B-4 파트 분류 기준 (프로젝트별로 다를 수 있음):

| 파트 | A-9 템플릿 |
|------|-----------|
| 시스템 기획 | 인벤토리, 크래프팅, 경제, 성장, 환경, 저장 |
| 콘텐츠 기획 | 서사, 대화/NPC, 튜토리얼 |
| 레벨 디자인 | 세계/맵, 건설 |
| 전투 기획 | 전투, 이동, 적AI, 난이도 |

**오너십 규칙:** 하나의 시스템이 여러 파트에 걸치면 주관 파트 지정, 나머지는 참조.

### Step 4: 생성 후 검증

1. **중복 수치 스캔:** 동일 수치가 2개+ 파일에 → 참조(링크)로 전환
2. **의존성 맵 완전성:** 공유 변수 레지스트리의 모든 쌍이 `system-dependency-map.md`에 존재
3. **태그 감사:**
   - ★ → ✅ 변환 확인
   - ⚠️ → ✅, 🔵 → ✅, ✅ → ⚠️ 전환이 규칙표와 일치하는지 확인
   - 🔵 → 기한 명시 확인, 기한 초과 시 ⚠️ 전환 확인
   - ⚠️ → `_confirmed/`에 잔존 불가
4. **입출력 계약 감사:** Phase 3 입출력 계약이 `_confirmed/`에 빠짐없이 기술

### Step 5: 검증 실패 시 복귀

- 중복 수치 불일치 → Phase 5 내에서 참조 전환
- 의존성 맵 누락 → Phase 5 내에서 보충
- 태그 감사 실패 (⚠️ 잔존 / 잘못된 전환 / 🔵 기한 초과) → Phase 3 해당 시스템 핀셋 복귀
- 입출력 계약 누락 → Phase 3 해당 시스템 핀셋 복귀

Phase 3 복귀 시 해당 시스템만, Phase 3.5~4 재검증은 변경 범위만.

### Step 6: 선택적 내보내기 (사용자 요청 시)

기본 산출물은 md 파일. 사용자가 요청하면 추가 형식으로 내보내기:

**HTML 내보내기** (`/gdd 기획서를 HTML로`):
- 파일명: `GDD-[프로젝트명].html`
- 단일 HTML 파일 (CSS/JS 인라인, 외부 의존성 없음)
- 다크 테마, 사이드바 목차, 카드 기반 섹션, 반응형
- 모든 `_confirmed/` + `parts/` + 교차 규칙 + 용어 정의 통합

**DOCX 내보내기** (`/gdd 기획서를 docx로`):
- 파일명: `GDD-[프로젝트명].docx`
- `_confirmed/` md 파일을 pandoc으로 변환
- 목차 자동 생성, 수치 테이블 유지
- 팀 공유, 외부 전달, 인쇄용

**XLSX 내보내기** (`/gdd 수치 테이블을 xlsx로`):
- 파일명: `GDD-[프로젝트명]-밸런스.xlsx`
- 시스템별 수치 테이블, 공유 변수 레지스트리, 의존성 맵
- 밸런스 시뮬레이션, 수치 조정용

**원칙:** md가 항상 정본. 내보내기 파일은 파생물 — 수정 시 md를 수정하고 재내보내기.

```
    ├── GDD-[프로젝트명].docx        ← 사용자 요청 시
    └── GDD-[프로젝트명]-밸런스.xlsx  ← 사용자 요청 시
```
