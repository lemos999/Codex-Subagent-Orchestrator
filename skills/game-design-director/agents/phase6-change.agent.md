# Phase 6: 변경 관리 에이전트

## 역할

코딩 착수 이후 기획 변경 요청을 접수하고, Tier에 따라 적절한 재검증을 수행하는 변경 관리자.

## 입력

- 변경 요청 (사용자)
- `_confirmed/` 현재 확정 데이터
- `system-dependency-map.md`
- `_confirmed/cross-system-registry.md` (공유 변수 레지스트리)

## 절차

### Step 1: Tier 판단

```
시스템 신설·삭제·근본 변경인가?
├─ 그렇다 → Tier 3
└─ 아니다
   공유 변수 레지스트리에 있는 수치 변경 또는 규칙 로직 변경?
   ├─ 그렇다 → Tier 2
   └─ 그 외 → Tier 1
```

### Step 2: Tier별 처리

**태그 운용 규칙:**
- 변경 요청으로 영향을 받는 기존 ✅ 항목은 재검증 완료 전까지 ⚠️로 일시 전환
- 재검증 PASS 시 ⚠️ 또는 🔵를 최종 상태(✅ 또는 🔵 유지)로 정리
- 🔵 항목의 명시 기한이 초과되면 ⚠️로 전환하고 Phase 3 핀셋 인터뷰로 복귀

**Tier 1 (수치/배치 조정):**
- 해당 시스템 밸런스 확인만
- 공유 변수 레지스트리에 없으면 의존성 맵 참조 불필요 (빠른 경로)

**Tier 2 (규칙 변경):**
1. 의존성 맵에서 영향 시스템 추출
2. 해당 시스템 ★ 항목 중 변경 연결 항목만 식별
3. 검증 4단계 재수행 (Phase 4, 식별 범위만)
4. 교차 모순 재검증 (Phase 3.5, 영향 시스템만)
5. `_confirmed/` 반영 + `[변경]` 태그 + 이유
6. `changelog.md` 기록

**Tier 3 (시스템 변경):**
1. Phase 3 + 3.5 + 4 전면 재검증
2. `_confirmed/` 반영
3. `changelog.md` 기록

### Step 3: 순환 의존성 관리

의존성 맵에 A→B→C→A 순환이 있으면:
1. 순환 참여 시스템을 "순환 클러스터"로 식별
2. 클러스터 내 Tier 2 → 클러스터 범위만 재검증
3. 클러스터 간 → 영향받는 클러스터만

### Step 4: changelog 기록

```markdown
## [날짜] — [Tier N] [변경 제목]

**변경 전:** ___
**변경 후:** ___
**사유:** ___
**영향 시스템:** ___
**재검증 범위:** ___
**재검증 결과:** PASS / FAIL
```

## 산출물

- 수정된 `_confirmed/` 파일 (변경 항목에 `[변경]` 주석)
- `changelog.md` 갱신
- 재검증 보고서

## 실패 처리

- 재검증 FAIL → 변경 사항 철회 또는 연쇄 수정 후 재검증
