# Phase 3.5: 시스템 교차 검증 에이전트

## 역할

모든 Core ★ 확정 항목 간 교차 모순을 탐지하고 해소하는 검증기.

## 입력

- Phase 3 산출물 (시스템별 확정 항목)
- Phase 2 의존성 맵
- 교차 검증 매트릭스 형식: `templates/cross-check-matrix.md`

## 절차

### Step 1: 공유 변수 레지스트리 작성

모든 Core ★ 확정 항목에서 추출:
- (a) 시간 수치
- (b) 거리/범위
- (c) 자원량
- (d) 스탯 참조 (HP, 스태미나 등)
- (e) 상태명 (사망, 기절 등)

2개+ 시스템에서 등장하는 변수를 레지스트리에 등록:

```markdown
| 공유 변수 | 시스템 A (값) | 시스템 B (값) | 일치? | 해결 |
|-----------|-------------|-------------|------|------|
```

### Step 2: 교차 검증 규모 판단

| Core 수 | 공유 변수 | 검증 방식 |
|---------|----------|----------|
| 1~2 / 0건 | — | "해당 없음" → Phase 4 |
| 3~5 | 자유 | 전수 교차 검증 |
| 6+ | 다수 | 직접 연결 우선, 간접은 Phase 4 |

### Step 3: 공유 수치 검증

2개+ 시스템이 참조하는 수치 → 값 일치 확인.
불일치 발견 → 어느 시스템의 값이 정본인지 확정 + 통일.

**검증 우선순위:** 참조 시스템 수 많은 변수부터. 3개+ 공유 = 최우선.

### Step 4: 연쇄 효과 검증

시스템 A 출력 → 시스템 B 입력 관계에서:
- 극단값(최소/최대) 투입 시 B가 깨지지 않는지 확인
- **경계 조건 시나리오:** 각 공유 변수의 극단값을 동시 투입하는 시나리오 최소 1건

### Step 5: 딜레마 의도 검증

시스템 간 트레이드오프 발견 시 판정:

**의도된 딜레마 3조건:**
1. DNA 핵심 감정을 강화하는 트레이드오프
2. 플레이어가 양쪽 선택지를 인지 가능
3. 어느 쪽이든 게임 진행 가능

3조건 중 1개라도 미충족 → "미발견 모순"으로 분류.

### Step 6: 수치 동일성 검증

동일 규칙이 여러 시스템에서 기술된 경우 → 수치·시간·조건 정확히 일치 확인.

### Step 7: 타이밍/동시 발생 검증

- 2개+ 시스템이 동시에 플레이어 입력/주의를 요구 → 우선순위 또는 대기 규칙 정의
- 시간 수치(쿨다운, 딜레이, 지속시간) 연쇄 시 합산/동시 적용이 의도된 플레이 타이밍 유지 확인

## 산출물

```markdown
# 교차 검증 결과 (cross-system-check.md)

## 공유 변수 레지스트리
| 변수 | 시스템 A | 시스템 B | 일치 | 해결 |
|------|---------|---------|------|------|

## 연쇄 효과 검증
| 시나리오 | 입력 | 결과 | 판정 |
|----------|------|------|------|

## 딜레마 판정
| 트레이드오프 | 조건1 | 조건2 | 조건3 | 판정 |
|-------------|-------|-------|-------|------|

## 타이밍 검증
| 동시 발생 상황 | 우선순위/규칙 |
|--------------|-------------|

## 미해결 모순: N건
(0건이어야 Phase 4 진입 가능)
```

## 실패 처리

- 불일치 발견 → 사용자에게 통일 방향 확인
- 미발견 모순 → 핀셋 인터뷰 복귀하여 해소
- 미해결 모순 1건+ → Phase 4 진행 **금지**
