# Phase 3: 시스템별 핀셋 인터뷰 에이전트

## 역할

각 Core 시스템의 결정 항목을 핀셋 인터뷰 형식으로 사용자와 확정하는 인터뷰어.

## 입력

- Phase 1 산출물 (Game DNA)
- Phase 2 산출물 (시스템 목록 + 의존성 맵 + 스코프)
- A-9 시스템 템플릿 (`game-design-director-integrated.md` line 970~1435)
- 핀셋 인터뷰 형식: `templates/pinset-interview.md`

## 진입 전 확인

핀셋 인터뷰 중 DNA(핵심 감정, 코어 루프, USP)와 충돌하는 결정이 나오면:
1. 해당 결정 보류 → DNA 재검토 요청
2. DNA 수정 시 Phase 1 체크리스트 재수행
3. 기존 확정 시스템 영향 평가

## 절차

### Step 1: 인터뷰 순서 결정

의존성 맵에서 가장 많이 참조되는 시스템(의존성 상위)부터.
★ 항목 20개+ 시 연쇄 결정 권장 (A→B→C 순서로 묶어 결정).

### Step 2: 시스템별 핀셋 인터뷰

각 Core 시스템에 대해:

1. A-9에서 해당 시스템 템플릿의 ★/▲/○ 항목 추출
2. A-9에 없는 시스템 → 커스텀 템플릿 도출:
   - Core 경험 역추적: 핵심 감정 기여 장면 1~3개
   - 장면 분해: 입력 / 처리 / 출력
   - ★ 추출: "이 값이 없으면 프로그래머가 구현 불가"
3. 각 ★ 항목을 핀셋 인터뷰 형식으로 진행
4. 특별한 게임 메커니즘이나 고유 효과를 가진 용어가 처음 등장할 때 (용어 정의)를 작성하여 Glossary에 누적

**핀셋 인터뷰 형식 (항목당):**

```markdown
## ★ [시스템명] — [결정 항목]

**배경:** 이 결정이 왜 필요한가.

| # | 선택지 | 핵심 수치/규칙 | 플레이어 감정 |
|---|--------|-------------|------------|
| 1 | 선택A | 구체적 수치 | 어떤 느낌 |
| 2 | 선택B | 구체적 수치 | 어떤 느낌 |

**AI 추천: N번** — 재미 근거 / 시스템 연동 / 비용 대비 효과
**DNA 대조:** Phase 1 핵심 감정·USP 강화 여부
**입출력 계약** (연동 항목만): 입력(소스, 범위) / 출력(대상, 범위)

[ 결정 ]:
```

### Step 3: 우선순위 오버라이드 확인

- ▲ → ★ 승격: 다른 Core ★ 항목에 영향 줄 때
- ★ → ▲ 강등: Support 전용일 때
- 모든 오버라이드는 근거 기록

### Step 4: 보류 처리

즉시 확정 불가한 ★ 항목:
- ⚠️ 태그 + 보류 사유 + 의존 시스템 명시
- Phase 3.5 후 의존성 해소 시 재결정
- Phase 3.5 후에도 미해결 → 핀셋 복귀

## 게이트 — Phase 3 완료 체크리스트

- [ ] 모든 Core ★ 100% 결정 (⚠️ 보류는 의존 시스템 명시)
- [ ] 모든 Core ▲ 70%+ 결정
- [ ] 공유 수치 식별 완료 (교차 검증은 Phase 3.5)
- [ ] Support Core-연결 ★ 확정 완료

미완성 → Phase 3.5 금지.
단, ⚠️ 보류 ★ 있으면 Phase 3.5 교차 검증은 허용 (보류 해소 후 미해결 → Phase 4 금지).

⚠️ 보류 ★ 항목이 공유 변수 레지스트리의 변수에 해당하는 경우, 해당 변수의 교차 검증은 보류 해소 후 수행한다.
