## 공통 원칙 (A-2)

- **재미가 최우선.** 모든 결정은 "왜 재미있는가?"로 정당화
- **모호한 결정 = 미완료.** 수치/규칙/조건 중 적합한 형태로 명시
  - 수치: "빠른 전투" ✗ → "공격 딜레이 0.3초" ✓
  - 규칙: "빛이 반사됨" ✗ → "입사각 = 반사각, 표면 법선 기준" ✓
  - 조건: "나쁜 결말" ✗ → "NPC 생존 0건 + 도덕 < 30 → 엔딩 C" ✓
- **What만 정의, How는 개발자 판단.** 엔진 중립 언어만 사용
- **선택에는 희생이 따른다.** 시간·자원·리스크 중 하나를 소모

## 재미 판단 4문항 (A-3)

1. **압박 (Tension):** 해결할 문제가 있는가?
2. **대응 (Agency):** 의미 있는 선택으로 해결 가능한가?
3. **피드백 (Feedback):** 결과가 즉각·명확한가?
4. **성장 (Mastery):** 반복할수록 더 잘할 수 있는가?

4개 ✅ → Core / 2~3개 → Support / 1개 이하 → 필요성 재검토

(이론 배경·적용 예시·약점 패턴: → 프레임워크 원본 A-8 재미 이론 참조서)

## 검증 4단계 (A-4)

순차 수행. 어느 단계에서든 FAIL → 정지 → 핀셋 인터뷰 복귀.

| 단계 | 검증 항목 | PASS 기준 | FAIL 기준 |
|------|----------|----------|----------|
| 1. 오류 | 누락 수치, Null, 0 나눗셈 | ★ 항목 수치 100% 기입 | ★ 미정의 1건+ |
| 2. 모순 | 규칙 충돌, 익스플로잇, 의도하지 않은 루프 | Phase 3.5 미해결 0건 + 극단값 파괴 없음 + 의도하지 않은 양성 피드백 루프 0건 | 미해결 1건+ |
| 3. 맥락 | 세계관, 고증, 톤앤매너 | 세계관 위반 0건 | 위반 1건+ |
| 4. 구현 | 소설 언어, 엔진 API, 예외 분기 | 소설 언어 0건 + 엔진 API 0건 + 예외 3개 이상 시 재설계 | 위반 1건+ |

**프로젝트별 정량 검증 훅:** Part B에 도메인 특화 정량 검증 기준이 있으면 Phase 4 2단계(모순)에서 통합한다.

**익스플로잇 판정 가이드:** 익스플로잇은 "설계 의도를 우회하는 플레이어 행동"이다. 장르별 예시: 전투/경제는 무한 자원·무한 스턴, 퍼즐은 의도하지 않은 해법, 시뮬레이션은 경제 무한 성장 루프.

**피드백 루프 검증:** 시스템 간 순환 관계가 있으면 양성 피드백 루프(강화)와 음성 피드백 루프(안정화)를 구분한다. 의도된 양성 루프는 딜레마, 의도하지 않은 양성 루프(죽음의 나선·무한 성장)는 모순으로 처리한다.

## 기획서 작성 표준 (A-5)

- **Step-by-Step:** ① 입력 → ② 시청각 피드백 → ③ 상태 제어 → ④ 연산 → ⑤ 결과·해제
- **오버엔지니어링 3문항:** (1) 비개발자가 이해 가능? (2) 구현 자유 침해 안 함? (3) 핵심 재미 위한 규칙?
- **형식화 수준:** 서술(행동 규칙) / 도표(스탯, 확률) / 순서도(상태 전이) — 모호하면 승격
- **시스템 인터페이스:** 타 시스템 연결 ★ 항목은 입력(값+범위) / 출력(값+범위) / 실패 시 기본값 명시
- **태그:** ★ 코딩 전 필수 / ▲ 1~2주 내 / ○ 개발 중 가능
- **용어 정의:** 특별한 게임 메커니즘이나 고유 효과를 가진 단어가 처음 등장할 때 반드시 (용어 정의)를 포함한다. 용어 정의는 Phase 3 핀셋 인터뷰에서 누적하여 Glossary 섹션으로 관리한다.

## Phase 1: 게임 DNA 확정

**에이전트:** `agents/phase1-dna.agent.md`
**상호작용:** 필수 (사용자 응답 필요)

사용자에게 5가지를 순서대로 확인:
1. **핵심 감정** — 주 감정 1개 (부 감정 1개 선택적, 주 감정 강화 방향만)
2. **코어 루프** — 마이크로(30초~3분) / 미들(5~30분) / 매크로(1시간~세션)
3. **장르 컨벤션** — 기대 요소 5~7개 + 의도적 제거 항목
4. **USP** — "[장르]인데 [다른 점] 때문에 [새로운 경험]"
5. **타겟 플레이어** — 연령대, 세션 길이, 실패 허용도, 환경

**게이트 — DNA 일관성 검증:**
- [ ] 핵심 감정 ↔ 코어 루프 시간 압박 양립?
- [ ] USP ↔ 장르 컨벤션 포함/제외 모순 없음?
- [ ] 타겟 실패 허용도 ↔ 핵심 감정 일치?
- [ ] 3레이어 루프가 모두 핵심 감정 강화?
- [ ] 마이크로 → 미들 → 매크로 피드 연결?
- [ ] 핵심 장면이 마이크로/미들에서 반복 발생?

불일치 → DNA 수정 → 체크리스트 재수행. 미완성 → Phase 2 금지.

## Phase 2: 시스템 우선순위

**에이전트:** `agents/phase2-priority.agent.md`
**상호작용:** 반(사용자 확인 필요)

1. 게임에 필요한 시스템 전체 나열
2. Core/Support 분류 (재미 4문항 기반, 메타 시스템은 첫 번째 기준만)

5. 스코프 어댑터 선택

**스코프 어댑터:**

| 스코프 | 팀 규모 | Core ★ | Support | Phase 3.5 |
|--------|---------|--------|---------|-----------|
| 린 | 1~3인 | 전수 | ★만 | 공유 변수만 |
| 표준 | 4~15인 | 전수 | ★/▲ | 전체 매트릭스 |
| 풀 | 16인+ | 전수 | 전수 | 전체 + 정량 |

시스템 복잡도 보정: Core 8개+ → 최소 표준, Core 12개+ → 풀

**게이트:** 모든 시스템 나열·분류 완료 + 의존성 맵 초안 + 스코프 선택. 미완성 → Phase 3 금지.

**인터뷰 순서:** 의존성 상위 시스템부터 (★ 항목 20개+ 시 연쇄 결정 권장)

## Phase 3.5: 시스템 교차 검증

**에이전트:** `agents/phase35-crosscheck.agent.md`
**상호작용:** 최소 (자동 검증, 모순 발견 시만 사용자 판단)

1. **공유 수치 검증** — 2개+ 시스템 참조 수치 목록 → 값 일치 확인
2. **연쇄 효과 검증** — 극단값(최소/최대)에서 깨지지 않는지 확인
3. **딜레마 의도 검증** — 트레이드오프가 "의도된 딜레마"인지 "미발견 모순"인지 판정
4. **수치 동일성 검증** — 동일 규칙이 여러 곳에서 기술될 때 일치 확인
5. **타이밍/동시 발생 검증** — 동시 입력 요구 시 우선순위 정의

**공유 변수 레지스트리:** `templates/cross-check-matrix.md` 형식으로 작성

**교차 검증 규모 적응:**
- Core 1~2개 / 공유 변수 0건 → "해당 없음" → Phase 4
- Core 3~5개 → 전수 검증
- Core 6개+ → 직접 연결 우선, 간접은 Phase 4에서

산출물: `cross-system-check.md`. 미해결 모순 0건 → Phase 4 진입.

## Phase 4: 재귀적 검증

**에이전트:** `agents/phase4-verify.agent.md`
**상호작용:** 최소 (자동 검증, FAIL 시 핀셋 복귀 안내)

검증 4단계(A-4) + 오버엔지니어링 3문항(A-5) 수행.

**"코딩 착수 가능" 게이트 (6조건):**
1. Core ★: 검증 4단계 통과 + 소설 언어 0건
2. Core ▲: 방향 확정 (🔵 허용, 기한 명시)
3. 교차 모순: 0건
4. Support Core-연결 ★: 확정 + 검증 1~2번 통과
5. 오버엔지니어링 3문항: 전 시스템 통과
6. 프로그래머 관점 셀프체크: (a)~(e) 전항목 확인

전부 통과 → "코딩 착수 가능" 선언. 미통과 → Phase 5 금지.

- ✅ → ⚠️ : Phase 6 변경 요청 접수 시, 재검증 완료까지 일시 전환
- ⚠️ 또는 🔵 태그가 남은 항목은 `_confirmed/` 진입 불가
- 🔵 기한 초과 시: 해당 항목을 ⚠️로 전환하고, Phase 3 핀셋 인터뷰로 복귀한다. 해당 항목이 Core 연결 항목이면 "코딩 착수 가능" 선언을 해당 항목 확정까지 보류한다.

# Mode B: 기존 게임 분석 (Step 1~4)

**에이전트:** `agents/modeb-analysis.agent.md`

**Step 흐름 요약:**

| Step | 입력 | 핵심 작업 | 산출물 | 게이트 |

| 진입 | 기존 자료 | 게임 상태·분석 목적·자료 범위 확인 | 분석 범위 확정 | 3항목 확인 |

# `/sub` 연동

Phase별 에이전트를 `/sub` orchestrator로 실행 가능.

**자동 실행 가능 Phase (사용자 입력 불필요):**
- Phase 3.5 교차 검증
- Phase 4 재귀적 검증
- Phase 5 문서 생성 (목록 확정 후)
- Mode B Step 1 진단

**사용자 상호작용 필수 Phase:**
- Phase 1 DNA 확정
- Phase 2 시스템 분류 확인
- Phase 3 핀셋 인터뷰

**하이브리드 파이프라인:**
Phase 1~3은 대화형으로 진행, Phase 3.5~5를 `/sub`로 자동화.

```
[사용자 대화] Phase 1→2→3
        ↓ (★ 확정 완료)
[/sub 자동] Phase 3.5 → Phase 4 → Phase 5
        ↓
[사용자 확인] 최종 산출물 리뷰
```

# 5축 재검토 기준

모든 판단의 귀결점. 우선순위: **빈틈 > 검증력 > 명확성 > 확장성 > 속도.**

| 축 | 질문 | 통과 |
|----|------|------|
| 빈틈 | 구멍이 생기는 지점? | 코딩 착수에 필요한 정보 100% 존재 |
| 속도 | 불필요한 중복/병목? | 더 적은 단계로 같은 품질 불가 |
| 명확성 | 읽고 즉시 실행 가능? | 추가 질문 없이 Phase 1부터 실행 가능 |
| 확장성 | 다른 장르에도 적용? | 3개+ 장르에서 절차 유지 |
| 검증력 | "코딩 가능" 확신? | 구현 중 "기획에 없는데?" 0건 |
