# 핀셋 인터뷰 형식

> 각 ★/▲/○ 결정 항목을 이 형식으로 사용자에게 제시한다.

---

## 기본 형식

```markdown
## ★/▲/○ [시스템명] — [결정 항목]

**배경:** 이 결정이 왜 필요한가. 이 항목이 미결이면 어떤 문제가 발생하는지.

| # | 선택지 | 핵심 수치/규칙 | 플레이어 감정 |
|---|--------|-------------|------------|
| 1 | 선택A | 구체적 수치 | 어떤 느낌 |
| 2 | 선택B | 구체적 수치 | 어떤 느낌 |
| 3 | 선택C | 구체적 수치 | 어떤 느낌 |

**AI 추천: N번**
- 재미 근거: ___
- 시스템 연동: ___
- 비용 대비 효과: ___

**DNA 대조:** 이 선택지가 Phase 1 DNA(핵심 감정·USP)를 강화하는가?
- 강화: ___
- 약화 시: 사유 명시 또는 대안 탐색

**입출력 계약** (타 시스템 연동 항목에 한함):
- 입력: [소스 시스템] → [값], 유효 범위 [min~max]
- 출력: [대상 시스템] → [값], 유효 범위 [min~max]
- 실패 시 기본값: ___

[ 결정 ]:
```

---

## 작성 규칙

1. **선택지는 대표 패턴.** 사용자가 테이블에 없는 선택도 가능. "기타: 직접 정의" 행 허용.
2. **수치는 구체적으로.** "빠름" ✗ → "0.3초" ✓
3. **감정은 핵심 감정과 연결.** Phase 1 DNA에서 정의한 핵심 감정과의 관계 명시.
4. **AI 추천은 근거 필수.** "좋을 것 같다" ✗ → "핵심 감정 '긴장'을 강화하며, 전투 시스템과 시너지" ✓
5. **DNA 대조는 항상 수행.** 약화하면 사유 없이 넘기지 않는다.
6. **입출력 계약은 연동 항목만.** 단일 시스템 내부 수치에는 불필요.

## 연쇄 결정 형식 (★ 20개+ 시)

의존성 A → B → C 관계의 ★ 항목을 묶어 진행:

```markdown
## 연쇄 결정: [시스템 A] → [시스템 B] → [시스템 C]

### ★ [시스템 A] — [항목]
(기본 형식과 동일)
[ 결정 ]: ___

↓ 이 결정이 아래 선택지를 좁힌다:

### ★ [시스템 B] — [항목]
(A 결정 반영된 선택지)
[ 결정 ]: ___

↓

### ★ [시스템 C] — [항목]
(A, B 결정 반영된 선택지)
[ 결정 ]: ___
```

각 항목의 DNA 대조와 입출력 계약은 개별 수행.

---

## ⚠️ 보류 형식

```markdown
## ⚠️ [시스템명] — [결정 항목]

**보류 사유:** [다른 시스템 X의 Y 항목이 확정되어야 결정 가능]
**의존 시스템:** [X]
**예상 해소 시점:** Phase 3.5 교차 검증 후
**임시 방향 (있으면):** ___
```

## 커스텀 템플릿 도출 (A-9에 없는 시스템)

A-9 22종에 해당하지 않는 시스템일 때:

1. **Core 경험 역추적:** 이 시스템이 핵심 감정에 기여하는 장면 1~3개
2. **장면 분해:** 각 장면의 플레이어 입력 / 시스템 처리 / 결과 출력
3. **★ 추출:** 처리 단계에서 "이 값이 없으면 프로그래머가 구현 불가"인 것
4. **핀셋 형식 적용:** 추출된 ★을 위 기본 형식으로 인터뷰

예시: 파티 시스템(JRPG) → "4인 파티 보스전 전략" → 편성/배치/스킬 → ★: 최대 인원, 전투 참여 인원, AI 행동, 대기열 방식
