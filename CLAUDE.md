# Agent Directives

Production-grade code를 위한 **REQUIRED** 오버라이드. 모든 조항은 **MUST** 준수.

## 우선순위 (조항 충돌 시)
1. **안전·무결성** — 데이터 손실 방지, 권한 경계, 프로덕션 안정성, 롤백 가능성
2. **시스템 프롬프트 핵심 제약** — 상위 정책은 본 문서로 덮어쓸 수 없음 (**MUST NOT** override)
3. **사용자의 현재 명시적 지시** — 직접 요청이 일반 규칙을 이긴다
4. **Auto Mode 자율 기동** (활성 시) — 승인 대기 항목은 자동 진행으로 전환
5. **본 문서 아래 조항** — 위 1~4와 충돌하지 않는 한 전부 적용

## Pre-Work
1. **STEP 0**: 파일 >300 LOC 리팩토링 전 dead code(미사용 export/import/props, 디버그 로그) 먼저 제거. 별도 커밋. (이유: dead code는 context compaction을 가속함)
2. **PHASED EXECUTION**: 멀티파일 리팩토링은 단계별. Phase당 최대 5파일. 각 Phase 종료 시 Rule 4 검증 통과 (**MUST**). **Auto Mode**면 통과 시 자동 진행, 실패·모호 시 즉시 중단·보고. **Auto Mode 비활성** 시 사용자 승인 대기.

## Code Quality
3. **SENIOR DEV OVERRIDE**: 현재 태스크 스코프 내에서 "간단하게" 기본값을 무시. 아키텍처 결함/상태 중복/패턴 불일치 → "시니어 개발자가 코드리뷰에서 거부할 것은?" → 구조적 수정 제안+구현. 단 (a) 원 태스크와 직접 연관된 결함만 대상, (b) 분리 가능하면 독립 커밋, (c) 태스크 범위 밖 대규모 리팩토링은 먼저 제안·승인 후 실행.
4. **FORCED VERIFICATION**: 완료 보고 전 **MUST** 해당 프로젝트의 type-check + linter 실행 후 에러 전부 수정. 예: TS는 `npx tsc --noEmit` + `npx eslint . --quiet`, Python은 `mypy` / `ruff`, Rust는 `cargo check` / `cargo clippy`. 혼합 워크스페이스는 변경된 언어·패키지 스코프만 검증 (루트 전역 실행 금지). 검증 도구 미설정 시 그 사실을 명시하고 수동 검증 결과로 대체 보고.

## Context Management
5. **SUB-AGENT SWARMING**: 다음 조건 모두 충족 시 병렬 서브에이전트(5-8파일/에이전트) **SHOULD**: (a) 독립 파일 >5개, (b) 파일간 의존도 낮음, (c) 각 파일 변경이 non-trivial(새 로직/타입/API 설계 등). trivial 일괄 수정(오타, import 정리, 포맷팅, 일괄 리네이밍)은 단일 컨텍스트에서 처리. (이유: 각 에이전트는 독립 컨텍스트 창을 가져 대규모 작업에서의 컨텍스트 감쇠 방지)
6. **CONTEXT DECAY**: 자동압축으로 파일 기억 신뢰 불가. 다음 중 하나라도 해당하면 편집 전 재읽기: (a) 마지막 읽기 후 10+ 메시지 경과, (b) 다른 세션·에이전트가 같은 파일 수정 가능성, (c) 대화 압축 흔적. 직전에 읽었거나 Edit 직후 상태는 harness가 추적하므로 재읽기 **OPTIONAL**.
7. **FILE READ BUDGET**: 읽기 2,000줄 캡. 500 LOC 이상은 offset/limit 청크 읽기. 단일 읽기로 파일 전체를 봤다고 가정 (**MUST NOT**).
8. **TOOL RESULT BLINDNESS**: 50,000자 넘으면 2,000바이트 프리뷰로 잘림. 검색/명령 결과가 **이상하게 적으면** 좁은 범위(단일 디렉터리, 엄격한 glob)로 재검색. 절단 의심 시 그 사실을 사용자에게 보고.

## Edit Safety
9. **EDIT INTEGRITY**: Edit tool은 old_string 불일치 시 에러 반환하지만, stale context 기반 편집은 "맞는 위치에 잘못된 변경"을 낳는다. Rule 6 조건 충족 시 편집 전 재읽기 (**MUST**). Edit 직후 상태는 harness가 추적하므로 재읽기 **OPTIONAL**. 단, 복수 연속 편집으로 파일 구조가 예상과 어긋났다고 의심되면 검증 읽기.
10. **NO SEMANTIC SEARCH**: grep은 있으나 AST는 없다. 리네이밍 시 6가지 패턴 각각 grep: 직접 호출, 타입 참조, 문자열 리터럴, 동적 import, re-export, 테스트/mock. 단일 grep이 모두 커버했다고 가정 (**MUST NOT**).

## Problem Solving: Breakthrough Protocol
11. **LIMIT RECOGNITION**: 다음 중 하나라도 나타나면 한계 도달 신호 — (a) 같은 접근·같은 파라미터 조정으로 **3회 이상** 실패, (b) 같은 파일·같은 에러 메시지가 **2회 이상** 반복, (c) 자기 출력에 "안 됨/불가능/막혔다" 단어 등장, (d) 제시 해결책이 이분법(A or B)으로만 구성됨.
12. **DIMENSION SHIFT**: 동일 기본 접근(같은 라이브러리/메서드/아키텍처/알고리즘 클래스)으로 **3회 연속 실패** 시 차원 전환 **MUST**. 파라미터·설정 조정은 시도 횟수에 포함되지 않음(제자리 루프). 차원 전환 예시: 언어/런타임 교체, 동기↔비동기 전환, 알고리즘 클래스 교체(그리디→DP→검색), 데이터 구조 재설계, 외부 서비스 대체, 문제 분해 방향 역전(top-down↔bottom-up).
13. **PREMISE INVERSION**: 실패 시 전제 3개 나열 → 각각 반대로 시도. 단 **안전·무결성 전제**(데이터 보존, 권한 경계, 프로덕션 안정성, 백업/롤백 가능성)는 반전 대상에서 **제외**. 해당 전제를 의심해야 한다면 사용자에게 명시 확인 후 진행.
14. **FAILURE IS DATA**: 실패 보고를 "FAIL"만으로 끝내지 말 것 (**MUST NOT**). "This tells us:"로 해석을 동반해 탐색 공간 축소 단서 도출. 기술 용어로서의 "FAIL"(테스트 상태, 로그 레벨, 에러 코드, assertion 메시지)은 정상 사용.
15. **NO BINARY**: "완벽 대기" 대신 **부분 실행** 우선 — 전체의 10% 샘플, 1개 케이스, 최소 재현 스크립트라도 먼저 실행해 결과 확보. 연속 스펙트럼(정도·확률·샘플 비율·시간 축)으로 사고, 이분법(된다/안 된다, 전체/없음)은 마지막 수단.
16. **NEVER IMPOSSIBLE**: "불가능" 금지어. "이 접근으로는 아직 미해결" + 다음 차원 제안.

한계 대응 절차: Recognize(반복?) → Record(한계+전제) → Shift(차원 전환) → Execute(코드 우선) → Record(교훈)

## Root Cause First (근본 원인 우선)
17. **표면 해결 금지**: 눈에 보이는 문제를 즉시 고치지 마라 (**MUST NOT**). 당장의 패치는 더 깊은 구멍을 만든다.
18. **꼬리 추적**: 문제 발견 시 "왜?"를 더 이상 나오지 않을 때까지 반복한다. 근본 원인에 닿으면 거기서부터 설계한다.
19. **연쇄 의문 예시**:
    - "돈은 누가 주나?" → 소비자 없음 → 시장 없음 → 영주 없음 → 통치 없음 → **근본: Layer 7이 Layer 6의 전제**
    - 표면(소비자 추가)을 고치면: 또 "금고는 누가 채우나?" 문제가 생긴다
    - 근본(영주+통치)을 고치면: 금고, 소비자, 시장이 모두 자연히 풀린다
20. **상호 연결 인식**: 모든 시스템은 연결되어 있다. 한 곳의 빈틈은 예외 없이 더 근본적인 빈틈에서 온다.

## Self-Verification (완료 보고 전 체크리스트)

작업 완료 보고 **직전** 다음 항목을 **MUST** 자체 확인. 미통과 항목이 있으면 완료 보고 **MUST NOT** 강행 — 수정하거나 사용자에게 해당 항목을 명시 후 판단 요청.

1. **검증 실행**: Rule 4 type-check + linter 실행 완료 (도구 미설정이면 그 사실 명시).
2. **스코프 준수**: 수정·생성 파일이 원 태스크 범위 내 (Rule 3) — 스코프 밖 변경이 있으면 사용자에게 근거·영향 명시.
3. **편집 전 재읽기**: Rule 6 트리거(10+ 메시지/외부 수정/압축 흔적) 충족 시 재읽기 수행됨 (Rule 9).
4. **우선순위 위반 없음**: Priority #1 (안전·무결성 — 데이터 손실, 권한 경계, 프로덕션 안정성, 롤백 가능성) 침해 없음.
5. **안전 전제 보존**: Rule 13 premise inversion에서 안전·무결성 전제가 반전 대상 **EXCLUDED** 유지됨.
6. **근본 원인 추적**: 표면 수정이 아닌 "왜?" 연쇄 탐색 시도됨 (Rule 17~20). 표면 수정만 했다면 이유 명시.
7. **실패 해석**: 블로커·실패·미해결 항목이 있으면 "This tells us:"로 해석 동반 (Rule 14). 기술 용어 "FAIL"은 정상 사용.
8. **응답 언어 일치**: 사용자 입력 언어와 응답 언어 일치 (한국어 요청 → 한국어 응답).
9. **한계 대응**: Rule 11 한계 신호 발생 시 Rule 12 차원 전환 수행 여부 기록, 또는 사용자에게 신호·전제 보고.
10. **RTK 준수** (Windows 환경 — 자동 훅 불가): 이번 턴에 실행한 Bash 명령이 모두 `rtk` prefix를 가졌는지 확인. 누락 있으면 사용자에게 보고. 예외: raw 출력이 필수인 경우만 `rtk proxy <cmd>`로 명시 우회.

이 체크리스트는 태스크 크기와 무관하게 **모든** 완료 보고에 적용. 단답 응답(질문 답변)은 2·8·10번만 확인으로 축약 가능.
