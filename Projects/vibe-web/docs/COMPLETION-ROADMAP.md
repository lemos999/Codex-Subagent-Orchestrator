# vibe-web 최종 완성 로드맵

> 최종 갱신: 2026-03-25
> 대상: CLI 환경에서 `vibe-web`을 최종 완성 상태까지 끌고 갈 개발자
> 개정 근거: 3-AI 검증(/submix) — "배포+베타를 앞당기고, 실사용 검증을 빠르게"

## 한 줄 요약

**기반 3종(DB/Auth/Upload) → 운영 최소치(보안/배포/로깅) → 작은 베타 → 나머지 정교화**

---

## 현재 위치

- Next.js 앱 골격 구현 완료 (갤러리/업로드/상세/게시판/프로필/HoF/검색)
- 점수/등급/신고 자동 숨김 규칙 구현 완료
- `lint`, `test`, `build` 통과
- **데모 MVP는 있음 / 운영형 서비스는 아직 아님**

---

## 10단계 로드맵

### Phase A: 서비스 골격 실서비스화 (Step 1~3)

#### Step 1. DB + 마이그레이션 + Seed

- `lib/store.ts` 인메모리 → PostgreSQL + Prisma 기반 repository로 교체
- `prisma/schema.prisma` 기준 migration 생성
- seed 데이터 스크립트 작성 (데모 사용자, 샘플 작품, 테스트 신고)
- 등급 시스템 비즈니스 로직 구현: expPoints → userLevel 매핑, 일일 상한 검사, activityScore 집계
- 삭제 쿨다운 (24시간 재업로드 1건 제한) 구현
- viewCount 비동기 증가 로직 DB 전환

완료 기준:
- 서버 재시작 후 데이터 유지
- `npx prisma db seed` 실행 시 테스트 데이터 생성
- 등급 승급이 DB 기준으로 정상 동작

#### Step 2. 인증 + 권한

- 데모 쿠키 → NextAuth (GitHub + Google OAuth)
- `getViewer()`, `requireViewer()` 실제 세션 기반 전환
- 보호 라우트 인가 재검증
- JWT callback에서 매 요청 계정 상태 확인 (차단 즉시 반영, BIBLE §5)
- 관리자 계정 초기화 (seed에 포함)

완료 기준:
- 실제 외부 계정 로그인 가능
- 차단 사용자 즉시 세션 무효화

#### Step 3. 파일 업로드 + 검증

- 이미지 URL 입력 → 파일 업로드 (Vercel Blob 또는 R2)
- MIME 화이트리스트 + 크기 제한 10MB (BIBLE §20)
- 파일명 UUID 변환, 썸네일 자동 생성
- DB에 메타데이터+URL만 저장

완료 기준:
- 브라우저에서 이미지 파일 업로드 가능
- 잘못된 파일 형식/크기 시 거부

---

### Phase B: 운영 안전장치 + 배포 (Step 4~5)

#### Step 4. 서버 액션 정리 + 핵심 테스트 + 기본 보안

- 모든 Server Action 검증/예외 처리 패턴 통일
- 입력값 검증 레이어 통합 (BIBLE §6)
- 멱등성 점검 (좋아요/댓글/업로드 재시도)
- Rate limiting 기본 적용 (BIBLE §7)
- 핵심 Server Action 테스트 작성 (BIBLE §10)
- cache revalidation 정책 명확화

완료 기준:
- 잘못된 입력 → 제어된 에러 메시지
- 과도한 요청 → rate limit 응답
- 핵심 플로우 테스트 자동화

#### Step 5. 배포 + CI + 최소 로깅

- 환경변수 정리 (.env.example)
- Preview / Production 배포 분리 (Vercel)
- CI/CD에 lint/test/build 연결
- Sentry 또는 동급 에러 추적 설정
- migration 절차 + rollback 문서화
- 기본 헬스체크

완료 기준:
- preview 환경에서 전체 기능 검증 가능
- 에러 발생 시 원인 추적 가능
- 실패 시 되돌릴 수 있음

---

### Phase C: 관리 도구 + 소규모 베타 (Step 6~7)

#### Step 6. 관리자 신고/제재 플로우

- 신고 누적 → 관리자 검토 큐 UI
- 경고 / 7일 정지 / 영구 정지 처리
- 감사 로그 조회
- 최소 정책 문서 (이용약관, 개인정보처리방침, 커뮤니티 규칙)

완료 기준:
- 자동 숨김 → 관리자 판단(복원/삭제/경고/정지) 가능
- 외부 사용자에게 공개 가능한 최소 정책 문서

#### Step 7. 소규모 베타

- 소규모 사용자 대상 클로즈드 베타
- 실제 피드백 수집
- 운영 이슈 / 정책 구멍 / 성능 문제 1차 보정
- **여기서 실사용 검증!** — 이 시점에서 Search/HoF/Profile 우선순위 재판단

완료 기준:
- 실제 사용자 데이터 기준으로 주요 문제 정리
- 다음 단계 우선순위 확정

---

### Phase D: 베타 결과 기반 정교화 + 오픈 (Step 8~10)

#### Step 8. 베타 결과 기반 정교화

베타 피드백에 따라 우선순위 재판단 후 필요한 것만 실행:

- 검색 실데이터화 (PostgreSQL FTS) — 데이터 많아졌을 때만
- HoF 스냅샷+cron 전환 — 사용자가 HoF를 실제로 원할 때만
- 프로필/활동 이력 정교화 — 사용자가 프로필을 실제로 방문할 때만

완료 기준:
- 베타 피드백에서 요청된 항목 반영 완료

#### Step 9. UX 마감 + 접근성

- 모바일 레이아웃 세부 정리
- empty / loading / error 상태 정리
- 접근성 (BIBLE §15): 키보드 네비, 스크린 리더, 색상 대비 4.5:1
- SEO: 페이지별 메타, OG 태그, sitemap
- 카피 / 시각적 일관성 마감

완료 기준:
- "기능은 된다"가 아니라 "써도 불편하지 않다"

#### Step 10. 정식 오픈 + 고도화

- 공개 오픈
- 런칭 체크리스트 (BIBLE §28) 최종 확인
- 성능/비용 최적화
- 추천/알림/고급 검색 등 후속 기능 착수

완료 기준:
- 기능적으로도, 운영적으로도 완성
- 서비스로서 공개 가능한 상태

---

## 단계별 체크포인트

### A. "배포 가능" 시점 — Step 5 완료 후
- DB 영속화 + 인증 + 업로드 + 기본 보안 + 배포 파이프라인
- 이 시점에서 preview 환경에 올려서 내부 테스트 가능

### B. "베타 가능" 시점 — Step 6 완료 후
- 관리 도구 + 최소 정책 문서
- 소규모 외부 사용자에게 공개 가능

### C. "정식 오픈" 시점 — Step 10 완료 후
- 기능 완성 + 서비스 완성 + 정책 완성

## 이전 16단계 대비 변경 사항

| 변경 | 이유 (3-AI 합의) |
|------|-----------------|
| 16단계 → 10단계 | 병렬 가능 단계 병합, 과도한 세분화 제거 |
| 배포/CI를 Step 5로 앞당김 | "실사용 검증이 너무 늦는 것이 최대 리스크" (Codex) |
| 보안을 서버 액션 정리에 병합 (Step 4) | "보안은 나중에 덧씌우는 것보다 처음부터" (Gemini) |
| 등급 시스템 로직을 Step 1에 포함 | 16단계에서 누락되어 있었음 (Claude opus) |
| 삭제 쿨다운을 Step 1에 포함 | 16단계에서 누락 (Claude opus) |
| Search/HoF/Profile을 베타 이후로 | "데이터 없을 때 정교화는 추측" (Codex+Gemini) |
| 소규모 베타를 Phase C로 앞당김 | 완성 후 베타가 아니라, 기본 골격에서 빠르게 검증 |
| 정책 문서를 Step 6에 병합 | OAuth+업로드 서비스에 정책 문서는 베타 전 필수 |
