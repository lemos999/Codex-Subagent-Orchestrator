# Trading Value

`Trading Value`는 `ETHUSDT Perp`, `BTCUSDT Perp` 자동매매 시스템을 설계하고 검증하기 위한 문서 중심 프로젝트다. 이 README의 목적은 단순 소개가 아니라, 새 세션이나 새 기여자가 이 파일 하나만 읽어도 프로젝트의 맥락, 목표, 목적, 히스토리, 현재 기준 문서, 읽는 순서, 작업 방향을 즉시 복원할 수 있게 만드는 것이다.

## 1. TL;DR

- 이 프로젝트는 `ETHUSDT Perp`, `BTCUSDT Perp` 자동매매 시스템 설계 프로젝트다.
- 대상 심볼은 `ETHUSDT Perp`, `BTCUSDT Perp`이다.
- 핵심 전략은 상위 구조가 살아 있을 때의 롱 편향과, 하위 약세 정렬이 강할 때의 제한적 반등 실패 숏이다.
- 현재는 구현보다 문서와 설계가 먼저 정리된 단계다.
- 현재 전략 소스 오브 트루스는 `coin_strategy_spec_v2.md`다.
- 현재 상태/수명주기/복구 구조 소스 오브 트루스는 `auto_trading_state_machine_design.md`다.
- `coin_strategy_spec.md`는 초기 원본 문서이며 보존본이다.
- 새 세션은 이 README를 읽고, 이어서 `coin_strategy_spec_v2.md`, `auto_trading_state_machine_design.md` 순서로 읽으면 전체 맥락을 복원할 수 있다.

## 2. 프로젝트 목적

이 프로젝트는 아래 네 가지 목적을 가진다.

1. 상위 추세를 우선하는 `ETHUSDT`, `BTCUSDT` 자동매매 전략을 문서화한다.
2. 사람이 차트를 보고 해석하던 규칙을 코드로 옮길 수 있는 수준으로 정제한다.
3. 실전 운영에 필요한 주문 계약, 로그, 장애 복구, 검증 절차를 함께 정의한다.
4. 전략 아이디어와 검증 프로세스를 분리해 장기적으로 진화 가능한 시스템 구조를 만든다.

## 3. 최종 목표

이 프로젝트의 최종 목표는 단기 수익률만 높은 전략을 만드는 것이 아니다. 최종 목표는 아래와 같다.

1. 백테스트, 페이퍼, 실전이 같은 규칙 집합을 사용하도록 만든다.
2. 상위 구조 편향, 명시적 무효화, 일관된 리스크 관리를 갖춘 자동매매 시스템을 만든다.
3. 실전 체결 오차와 장애 복구까지 포함해 운영 가능한 구조를 만든다.
4. 특정 구간 최적화가 아니라 검증을 통해 진화 가능한 시스템을 만든다.

## 4. 현재 상태

현재 이 폴더는 구현 코드보다 문서가 먼저 정리된 설계 단계다. 즉, 지금의 핵심 산출물은 실행 코드가 아니라 아래 세 가지다.

1. 전략 규칙
2. 상태머신 구조
3. 프로젝트 맥락과 작업 방향

현재 이 프로젝트에는 실거래 엔진 구현체가 아직 없다. 따라서 이 README는 새 세션이 "코드를 어디서 읽어야 하나"보다 먼저 "어떤 문서를 어떤 우선순위로 읽어야 하나"를 알게 만드는 것을 목표로 한다.

## 8. 소스 오브 트루스 규칙

문서 간 충돌이 보이면 아래 우선순위를 따른다.

1. 전략 규칙 충돌: `coin_strategy_spec_v2.md`를 우선한다.

`coin_strategy_spec.md`는 보존본이고, `coin_strategy_spec_v2.md`가 현재 전략 기준이며, `auto_trading_state_machine_design.md`는 그 전략을 안전하게 실행하기 위한 상태 구조 기준이다.

- 구현 전에 파라미터 최적화부터 하는 접근

## 11. 현재 확정된 핵심 전략 방향

현재 프로젝트는 아래 세 전략만 범위 안에 둔다.

- `TREND_LONG`
- `PULLBACK_LONG`
- `REBOUND_SHORT`

전략 철학은 한 문장으로 요약하면 아래와 같다.

상위 구조가 살아 있을 때는 하위 조정에서만 롱을 하고, 하위 약세 정렬이 강할 때만 제한적으로 반등 실패 숏을 사용한다.

이 전략군은 모두 같은 순서를 따른다.

1. 상위 구조를 본다.
2. 방향 편향을 본다.
3. 유효한 지지/저항 존을 고른다.
4. 하위 타임프레임 트리거를 확인한다.
5. 손절과 목표를 먼저 확정한다.
6. 그 다음에만 주문을 낸다.

## 12. 현재 확정된 시스템 방향

현재 시스템 설계는 아래 원칙을 따른다.

1. 모든 전략 판단은 완전히 마감된 봉 기준으로만 수행한다.
2. 포지션이 열려 있으면 반드시 보호 손절 주문이 존재해야 한다.
3. 같은 심볼에 대해 한 번에 하나의 순포지션만 가진다.
4. 반대 방향 포지션으로 한 번에 뒤집지 않는다.
5. 시장 해석 상태와 포지션 상태를 섞지 않는다.
6. 재기동 후 로컬 상태와 거래소 상태가 다르면 즉시 중단 또는 격리한다.

## 14. 문서 히스토리

이 폴더는 아직 독립적인 git 히스토리가 충분히 쌓이지 않았기 때문에, 아래 히스토리는 커밋 로그가 아니라 문서 진화 히스토리다.

### 14.1 1단계: 초기 전략 문서 작성

처음에는 `coin_strategy_spec.md`가 작성되었다. 이 문서는 전략 철학, 타임프레임 구조, 롱/숏 시나리오, 무효화 규칙, 개발 체크리스트를 큰 그림 중심으로 정리했다.

이 단계에서 이미 좋은 점이 있었다.

- 상위 구조 우선이라는 편향이 명확했다.
- 진입보다 무효화를 먼저 본다는 철학이 있었다.
- 검증과 실전 전환 절차를 중요하게 다뤘다.

하지만 이 단계 문서는 사람에게는 이해되지만, 기계 실행과 백테스트 재현성 측면에서는 해석 여지가 남아 있었다.

### 14.2 2단계: 구현 리뷰

초기 문서를 구현 관점으로 다시 검토하면서 다음 문제가 드러났다.

- `Visible Range` 기반 프로파일은 자동화에 비결정적이었다.
- `강세`, `상단 거절`, `박스 중앙`, `유지`, `충분히 약함` 같은 표현이 정량화되지 않았다.
- 단일 `mode`가 시장 상태와 전략 라우팅 의미를 동시에 가져 혼선이 생길 수 있었다.
- 일부 허용 경로는 상세 전략 규칙 없이 등장했다.
- 주문 집행과 보호 손절 계약이 충분히 명시되지 않았다.

이 리뷰는 프로젝트 방향을 바꾼 것이 아니라, 원래 의도를 더 안전하고 재현성 있게 만들기 위한 정제 단계였다.

### 14.3 3단계: 상태머신 구조화

그 다음에 `auto_trading_state_machine_design.md`가 작성되었다. 이 문서는 전략 규칙을 상태 관점으로 분해했다.

핵심 메시지는 아래와 같다.

- 시장을 해석하는 상태가 있다.
- 전략 셋업을 추적하는 상태가 있다.
- 실제 포지션을 관리하는 상태가 있다.
- 시스템과 리스크를 관리하는 상태가 있다.

이 네 층을 섞지 않아야 자동매매가 운영 가능해진다.

### 14.4 4단계: v2 실행 스펙 정리

그 다음에 `coin_strategy_spec_v2.md`가 작성되었다. 이 문서는 현재 구현 기준 전략 스펙이다.

이 단계에서 들어간 핵심 변화는 아래와 같다.

- 고정 프로파일 윈도우 도입
- 정량화된 판정식 도입
- 전략 허용 집합 구조 도입
- 리스크 기준 통일
- 보호 손절과 재기동 복구 계약 명문화
- 수용 테스트와 검증 단계 명시

즉, v2는 전략 아이디어를 바꾼 것이 아니라, 아이디어를 운영 가능한 계약으로 바꾼 문서다.

## 17. 다음 작업 방향

3. 이벤트 기반 백테스트 러너 구현

2. 셋업 추적기 구현
3. 주문 집행 계약 구현
4. 포지션 상태머신 구현
5. 페이퍼 트레이드 검증

## 18. 새 세션이 절대 놓치면 안 되는 불변조건

아래 항목은 문서를 읽을 때 반드시 기억해야 한다.

1. 현재 전략 기준은 `coin_strategy_spec_v2.md`다.
2. `coin_strategy_spec.md`는 현재 구현 기준이 아니다.
3. 모든 판단은 마감봉 기준이어야 한다.
4. 보호 손절 없는 포지션은 허용되지 않는다.
5. 상태 해석과 포지션 관리는 분리되어야 한다.
6. 자동매매의 목적은 높은 백테스트 수익률이 아니라 재현 가능한 운영 시스템 구축이다.

## 19. 성공 기준

이 프로젝트는 아래를 만족해야 성공으로 본다.

1. 같은 데이터 입력에 대해 같은 상태와 주문 의사결정이 재현된다.
2. 백테스트, 페이퍼, 실전 로그를 비교할 수 있다.
3. 보호 손절 누락, 상태 불일치, 중복 진입 같은 운영 사고를 구조적으로 막는다.
4. 전략 성과를 레짐별로 분리 평가할 수 있다.
5. 수익률뿐 아니라 기대값, 최대 낙폭, 연속 손실, 실전 오차를 함께 관리한다.

## 20. 최종 요약

이 README는 이 프로젝트의 소개문이 아니라, 새 세션을 위한 컨텍스트 복원 문서다. 이 파일 하나만 읽어도 아래를 알아야 한다.

- 무엇을 만드는 프로젝트인지
- 왜 이 프로젝트가 존재하는지
- 어떤 문서가 기준인지
- 어떤 순서로 읽어야 하는지
- 초기 문서와 현재 기준 문서의 차이가 무엇인지
- 다음 작업이 무엇인지

그 다음 단계에서 실제 규칙과 구현 기준은 `coin_strategy_spec_v2.md`, 상태와 실행 구조는 `auto_trading_state_machine_design.md`를 읽으면 된다.
