# 포스리드 문서 개선안 종합

이 문서는 현재 검토된 두 문서의 목적 적합도를 기준으로, 어떤 식으로 개선 및 수정하면 좋은지와 그렇게 수정했을 때 기대되는 결과를 한곳에 정리한 종합본이다.

대상 문서:
1. `forcelead_README.md`
2. `novel-persona.md`

---

## 1. `forcelead_README.md` 개선안

### 문서 목적

후속 작업자가 이 프로젝트의 현재 정합 상태, 판단 우선순위, 승인 경계, 다음 작업 순서를 빠르게 파악하고 안전하게 이어받도록 돕는 인수인계 문서다.

### 어떻게 개선 및 수정하면 좋은가

1. 문서 맨 앞에 `3분 시작 요약` 블록을 추가한다.
수정 방식:
제목 바로 아래에 `현재 확정 캐논`, `보류 안건`, `다음 작업`, `절대 먼저 확정하면 안 되는 것`을 4줄로 요약한 블록을 넣는다. 현재 마지막의 한 줄 요약은 이 블록으로 흡수한다.

2. 승인 경계를 한 섹션으로 통합한다.
수정 방식:
`승인 없이 진행 가능 / 사용자 승인 필요` 2열 표를 만들어, 현재 승인 안건과 작업 방식 규칙의 승인 관련 문장을 한곳에 모은다.

3. `프로젝트 채팅 확정 사항`을 발췌 섹션으로 만든다.
수정 방식:
현재 README에 반영된 사용자 최신 지시 중, 문서 밖 대화에만 있던 내용을 5~8개 핵심 bullet로 정리해 별도 섹션으로 둔다.

4. 부록성 정보를 본문 뒤로 내린다.
수정 방식:
`파일별 현재 판정`, `추천 프롬프트`를 `부록 A/B`로 내리고, 본문에는 `무엇을 읽고 무엇을 먼저 손댈지`만 남긴다.

5. 권장 작업 순서를 더 실행 단위로 바꾼다.
수정 방식:
Step 2~4를 `먼저 패치할 문서`, `그 문서에서 잠글 항목`, `승인 받고 넘어갈 분기점` 중심으로 다시 쓴다.

### 그렇게 개선 및 수정하면 기대되는 결과

1. 새 작업자가 README 첫 화면만 보고도 지금 무엇을 믿어야 하는지와 다음에 무엇을 해야 하는지를 즉시 파악할 수 있다.
2. 승인 없이 진행해도 되는 일과 사용자 확인이 필요한 일이 분리되어, 무단 확정이나 과속 정리가 줄어든다.
3. 문서 바깥 채팅 맥락 의존도가 낮아져서 README 자체가 더 닫힌 인수인계 문서가 된다.
4. 본문이 가벼워져 온보딩 속도가 빨라지고, 세부 참고 정보는 필요할 때만 뒤에서 꺼내 쓰게 된다.
5. 후속 작업 순서가 문서 재작성 실무 단위로 보이기 시작해, 실제 작업 착수 비용이 낮아진다.

### 우선순위

1. `3분 시작 요약` 추가
2. 승인 경계 통합 섹션 신설

### 한줄 결론

좋은 재료는 충분하지만, README를 진짜 인수인계 문서로 만들려면 첫 화면 요약과 승인 경계 통합부터 먼저 손봐야 한다.

## 2. `novel-persona.md` 개선안

### 문서 목적

소설 구성 요소를 페르소나 단위로 분리해, AI와의 협업 작업 시 설정 충돌 방지, 개연성 점검, 등장 연결 관리를 일관되게 수행하기 위한 운용 프레임워크 문서다.

### 어떻게 개선 및 수정하면 좋은가

1. 섹션 10을 분리해 문체 학습 계획을 별도 문서로 독립시킨다.
수정 방식:
섹션 10 전체를 `novel-style-guide.md` 또는 `novel-style-learning-plan.md`로 분리하고, 본 문서에는 `문체 관리자 역할 정의 + 별도 문서 링크`만 남긴다. 현재 문체 방향은 3~5줄 요약으로 축약한다.

2. 템플릿 중복을 제거하고 단일 정규 템플릿으로 통합한다.
수정 방식:
섹션 6.5, 10.16, 11에 흩어진 입력/출력 형식을 하나로 합친 뒤, 가장 완전한 버전을 정규 템플릿으로 고정한다. 섹션 11은 삭제하거나 `섹션 6 참조`로 바꾼다. 템플릿 아래에는 최소 1개의 작성 예시를 붙인다.

3. `[제안]`에서 `[확정]`으로 전환되는 기준을 문서화한다.
수정 방식:
섹션 0 또는 섹션 4에 `사용자가 명시적으로 승인하고, 이 문서에서 태그를 [확정]으로 바꾼 항목만 운용에서 확정으로 처리한다` 같은 가안 확정 절차를 추가한다. 실제로 이미 굳어진 항목은 태그를 갱신한다.

4. 섹션 순서를 운용 진입 흐름 중심으로 재배열한다.
수정 방식:
`문서 성격 -> 핵심 운용 방식/입출력 템플릿 -> 페르소나 구분 -> 카드 항목 -> 역할 위계 -> 용어 통일 -> 대화 운영 원칙 -> 핵심 메모 -> 문체 관리자 요약` 순으로 재정리한다.

5. 등장 연결 관리자 섹션 경계를 더 분명히 한다.
수정 방식:
등장 연결 관리자를 별도 소절로 올리고, 그 하위에 대표 업무와 장치 분류가 붙도록 구조를 다시 묶는다. 관리 페르소나 목록은 상단 인덱스로 먼저 제시한다.

6. 마지막 메모 섹션은 통합하거나 삭제한다.
수정 방식:
섹션 12의 반복 내용을 섹션 0 또는 9로 흡수하고, 독립 섹션은 제거한다.

### 그렇게 개선 및 수정하면 기대되는 결과

1. 핵심 운용 프레임워크가 절반 이하 분량으로 압축되어 실제 작업 중 빠르게 참조 가능해진다.
2. 어느 버전의 템플릿을 써야 하는지 고민이 사라지고, AI 작업 요청 형식이 표준화된다.
3. 운용 중 `이게 확정인가 아닌가`를 다시 묻는 반복이 줄어들고, 태그가 의사결정 이력처럼 기능하게 된다.
4. 처음 문서를 여는 사용자가 개념보다 사용법을 먼저 이해하게 되어 학습 비용이 낮아진다.
5. 등장 연결 관리자 파트의 귀속 구조가 명확해져 역할 선택과 활용이 더 쉬워진다.
6. 문서 끝의 중복 요약이 사라지고, 문서 전체 구조가 더 단정해진다.

### 우선순위

1. 섹션 10 분리
2. 템플릿 중복 제거 및 통합

### 한줄 결론

운용 원칙은 잘 잡혀 있으나, 미착수 문체 학습 계획이 문서 절반을 점령하고 입출력 템플릿이 여러 벌로 흩어져 있어 당장 쓰기 어려운 문서가 되어 있다.

## 3. 공통 정리

두 문서 모두 방향은 맞다. 다만 현재는 `정보를 많이 담은 상태`에 가깝고, 앞으로는 `목적에 맞게 빨리 쓰이는 문서`로 바꾸는 정리가 필요하다.

공통적으로 먼저 좋아지는 지점은 다음과 같다.

1. 첫 화면에서 문서의 역할과 사용법이 바로 보이게 된다.
2. 확정 정보와 제안 정보의 경계가 선명해진다.
3. 실무자가 문서를 읽고 바로 다음 행동으로 넘어가기 쉬워진다.
4. 긴 부록성 정보와 핵심 운용 정보가 분리되어 관리가 편해진다.

---

## 4. 최종 우선순위

1. `forcelead_README.md`에 `3분 시작 요약`과 `승인 경계 통합 섹션`을 먼저 넣는다.
2. `novel-persona.md`에서 문체 학습 계획을 분리하고, 입력/출력 템플릿을 하나로 통합한다.

이 두 축부터 정리하면, 이후 문서별 세부 개선은 훨씬 안정적으로 진행할 수 있다.
