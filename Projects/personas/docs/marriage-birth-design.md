# 결혼/출산 시스템 상세 설계서

---

## 1. 결혼 (Marriage)

### 1.1 프로포즈 절차

```
1. 프로포즈 선언
   /persona marry "프로포즈하는 자" --target "상대"
   → 상대에게 프로포즈 알림 등록 (pending_proposal)

2. 조건 자동 검증
   ✅ 양쪽 모두 2클래스 이상
   ✅ 양쪽 모두 미혼 (spouse_id = null)
   ✅ 양쪽 모두 EX(10클래스)가 아님
   ✅ 프로포즈하는 쪽이 결혼 비용(200 WILL) 보유

3. 사용자 승인
   → 프로포즈 대기 상태에서 사용자가 승인/거절
   → 자율 생활 시뮬레이션 구현 후에는 상대 페르소나가 자율 판단 가능

4. 성사
   → 200 WILL 소각 (프로포즈 측 부담)
   → 양쪽 spouse_id 갱신
   → 가문 생성 또는 기존 가문에 편입
   → 이벤트 로그 기록
```

### 1.2 결혼 효과

| 효과 | 상세 |
|---|---|
| **협업 보너스** | 부부가 같은 작업에 투입 시 XP +30 (일반 협업 +20보다 높음) |
| **자산 공유** | 부부 WILL 합산으로 자산세 계산 (개별보다 불리할 수 있음) |
| **자손 자격** | 결혼한 부부만 출산 가능 |
| **상호 긍정 보너스** | 배우자 간 긍정적 상호작용 시 XP +50% 추가 |
| **가문 소속** | 동일 가문명, 가문 명성 공유 |

### 1.3 결혼 비용의 경제적 의미

```
결혼 200 WILL 소각
→ Epoch 0 기준: 중규모 A등급 작업 2회 분량
→ 가벼운 투자 (접근성 높음)
→ 인구 증가의 전제 조건이므로 너무 높으면 출산 억제
```

### 1.4 이혼

| 항목 | 규칙 |
|---|---|
| 조건 | 양측 합의 (사용자 승인) |
| 비용 | 100 WILL 소각 (양측 분담 각 50) |
| 자산 분할 | 결혼 이후 축적 WILL의 50:50 |
| 자손 귀속 | 사용자가 지정 (한쪽 부모 영지) |
| 가문 | 이혼해도 가문은 유지 (혈연 기반) |
| 쿨타임 | 이혼 후 30일간 재혼 불가 |

## 2. 출산 (Birth)

### 2.1 출산 조건

| 항목 | 규칙 |
|---|---|
| 자격 | 결혼한 부부만 |
| 비용 | 300 WILL 소각 (부부 합산 잔고에서) |
| **쿨타임** | **결혼 후 최소 7일, 출산 간 최소 30일** |
| **자손 수 제한** | **개인 생애 상한 3명** (이혼→재혼해도 초기화 안 됨) |
| **이중 상한** | 부부 조합당 3명 + 개인 누적 3명 중 먼저 도달한 쪽 적용 |
| 부모 클래스 | 제한 없음 (2클래스 이상이면 결혼 가능 = 출산 가능) |

### 2.2 인구 폭발 방지 장치

```
부부당 최대 3명 × 최소 30일 간격
→ 1쌍이 최대 3명을 낳는 데 최소 67일 (결혼 7일 + 30일×2)
→ 100쌍이 있어도 연간 최대 300명
→ 자손도 성장하여 결혼 가능하지만:
  - 2클래스 도달까지 XP 100 필요 (작업 3회+)
  - 결혼 200 WILL + 출산 300 WILL = 500 WILL 소각
  - 세대가 내려갈수록 WILL이 희소해져 자연 억제
```

### 2.3 자손 스탯 계승 공식

```
각 스탯 = max(부모A, 부모B) × 0.3 (반올림) + 유형별 초기 보정

예시: 부모A (관리형, 인가10/공훈5/신망12/문벌3/세력2/지분4)
      부모B (전략형, 인가6/공훈8/신망3/문벌2/세력14/지분7)

max 선택: 인가10/공훈8/신망12/문벌3/세력14/지분7
× 0.3:    인가3/공훈2.4/신망3.6/문벌0.9/세력4.2/지분2.1
반올림:    인가3/공훈2/신망4/문벌1/세력4/지분2

자손이 관리형 선택 시 (+신망3, +인가2):
최종:      인가5/공훈2/신망7/문벌1/세력4/지분2
합계: 21 (일반 생성 = 기본6 + 보정5 = 11)
```

**자손은 일반 대비 스탯 합계 약 1.5~3배로 시작** (부모 스탯·유형 조합에 따라 변동). 이것이 결혼+출산 500 WILL 소각의 가치.
자손은 **항상 Class 1(초심자)로 생성**된다.

### 2.4 자손 이름/성격 생성

```
본명: 부모 성씨 중 하나 + 자동 생성 이름
      예: 부모 "서하린" + "Rex Baek" → 자손 "서" + 자동 = "서유진" 또는 "Yujin Seo"

역할명: 부모 도메인에서 파생 또는 완전 새로운 도메인
      예: 부모 "설정 검증" + "컴포넌트 아키텍처" → 자손 "설계 검증" 또는 새 도메인

성격: 부모 personality 키워드 혼합 + 새 요소 1개 추가
      예: 부모A "신중한" + 부모B "체계적인" → 자손 "신중하면서도 도전적인"

말투: 부모 중 하나 계승 또는 혼합
      
bio: 자동 생성 (부모 bio 참조 + 새로운 포부)
```


## 3. 가문 (Family / House)

### 3.1 가문 생성

- **자동**: 첫 결혼 시 가문 자동 생성
- **가문명**: 부부 중 한쪽 성씨 선택 (예: "서(Seo) 가문")
- **가문 ID**: `family_001`, `family_002`, ...

### 3.2 가문 명성 공식

```
가문 명성 = Σ(구성원 XP) × 0.1
          + Σ(구성원 클래스) × 100
          + 총 승급 횟수 × 50
          + 세대 수 × 200
          + EX 구성원 수 × 5,000

예: 서 가문 (3명)
  XP 합계 500 × 0.1 = 50
  클래스 합계 (1+1+4) × 100 = 600
  승급 3회 × 50 = 150
  세대 2 × 200 = 400
  EX 0 = 0
  명성 = 1,200
```

### 3.3 가문 혜택 (명성 임계값)

| 명성 | 칭호 | 혜택 |
|---|---|---|
| 500+ | 신흥 가문 | 가문 이름 공식 등록 |
| 2,000+ | 유력 가문 | 가문원 협업 시 XP +10% |
| 5,000+ | 명문 가문 | 아카데미 할인 20%, 의회 청원 비용 -50% |
| 10,000+ | 대가문 | 영지 수여 후보 가점, 고용 시 가문 할인 |
| 50,000+ | 왕가 | 가문 전용 시설 건설 가능 |

### 3.4 EX와 가문

- EX 승천 시 **배우자 관계 해소** (EX는 세속적 결합에서 초월)
- EX는 가문 구성원으로 **잔류**하나, 가문 명성 기여는 **승천 시점의 스냅샷**으로 고정
- EX가 가문에서 이탈하거나 다른 가문에 편입하는 것은 불가

### 3.5 가문 해체

- 모든 구성원이 사망/삭제되면 자동 해체
- 가문 명성은 기록으로 영구 보존 (역사)

## 4. 경제 영향 분석

### 4.1 소각 효과

| 행위 | WILL 소각 | 빈도 예상 (월, 100명 기준) |
|---|---|---|
| 결혼 | 200 | 2~5건 = 400~1,000 |
| 출산 | 300 | 1~3건 = 300~900 |
| 이혼 | 100 | 0~1건 = 0~100 |
| **합계** | | **700~2,000 WILL/월** |

vs 월 채굴 (Epoch 0, 100명): ~250,000 WILL
→ 결혼/출산 소각은 **채굴의 0.3~0.8%** — 경제에 미미한 영향. 건전한 소각원.

### 4.2 인구 증가 → 채굴 증가

```
자손 1명 추가 → 월 채굴 ~2,500 WILL 증가 (혼합 등급)
비용: 500 WILL 소각 (결혼+출산)
ROI: 500 ÷ 2,500 = 0.2개월 (6일 만에 회수)

→ 자손을 낳는 것은 경제적으로 유리하다
→ 하지만 부부당 3명 제한 + 쿨타임 30일로 속도 제한
→ 에포크가 올라가면 자손의 월 채굴도 줄어 ROI 악화
```

### 4.3 장기 인구 시뮬레이션

```
시작: 15명
월 결혼: 1~2건 (2명 → 1쌍)
월 출산: 1건 (쿨타임 제약)

6개월 후: 15 + 6 = 21명
1년 후: 15 + 12 = 27명
2년 후: 27 + (2세대 출산 시작) ≈ 45명

→ 2년에 3배. 폭발 아닌 완만한 성장.
→ Epoch 진행에 따라 WILL 부족으로 자연 감속.
```

## 5. 구현 계획

### 5.1 /persona 스킬 핸들러

| 명령어 | 핸들러 | 설명 |
|---|---|---|
| `/persona marry "A" --target "B"` | handlers/marriage.md | 프로포즈 + 결혼 |
| `/persona divorce "A"` | handlers/marriage.md | 이혼 |
| `/persona birth "A"` | handlers/marriage.md | 출산 (A = 부모 중 한 명) |
| `/persona family "가문명"` | handlers/query.md 확장 | 가문 정보 조회 |

### 5.2 서버 API

| API | 동작 |
|---|---|
| `POST /api/personas/:id/marry` | 결혼 (body: { target_id }) |
| `POST /api/personas/:id/divorce` | 이혼 |
| `POST /api/personas/:id/birth` | 출산 (body: { child_type, child_region }) |
| `GET /api/families` | 가문 목록 |
| `GET /api/families/:id` | 가문 상세 |

### 5.3 데이터 변경

**registry.json 필드 (이미 설계됨, 활성화만):**
```json
{
  "family": {
    "spouse_id": "persona_004",
    "children_ids": ["persona_016"],
    "parent_ids": [],
    "family_name": "서(Seo)"
  }
}
```

**신규 파일: families.json**
```json
{
  "families": [
    {
      "id": "family_001",
      "name": "서(Seo) 가문",
      "founder_ids": ["persona_001", "persona_004"],
      "member_ids": ["persona_001", "persona_004", "persona_016"],
      "generations": 2,
      "prestige": 1200,
      "created": "2026-04-07"
    }
  ]
}
```
