# Deep-Thinking Tokens: 깊이 있는 추론을 이끌어내는 AI 활용 가이드

> 기반 논문: *Think Deep, Not Just Long: Measuring LLM Reasoning Effort via Deep-Thinking Tokens*  
> (arXiv:2602.13517, 2026.02 — Google & University of Virginia)

---

## 0. 논문의 논리 구조 (상세)

### 0.1 출발점: "더 길게 생각하면 더 잘한다"는 통념

LLM의 추론 능력을 끌어올리는 주류 전략은 **test-time compute scaling** — 추론 시점에서 더 많은 연산을 투입하는 것이었다. 구체적으로는 Chain-of-Thought(CoT) 프롬프팅을 통해 모델이 "생각의 과정"을 길게 출력하도록 유도하고, 이 출력 토큰 수를 추론 노력의 대리 지표(proxy)로 간주했다.

이 접근은 OpenAI(o1, o3), Anthropic(Claude 3.7/4), DeepSeek-R1, Qwen3 등 주요 reasoning 모델의 설계 철학이기도 했다. "토큰을 더 많이 쓰게 하면 성능이 올라간다"는 전제 위에 모델 훈련과 프롬프트 전략이 구축되어 왔다.

### 0.2 문제 제기: 길이는 추론 품질의 역지표다

그러나 최근 연구들이 이 전제에 균열을 냈다:

- **역U자 관계** (Wu et al., 2025): CoT 길이와 정확도 사이에 최적점이 존재하며, 그 이후로는 성능이 하락한다.
- **역스케일링** (Gema et al., 2025): 더 긴 추론 체인이 **체계적으로** 정확도를 떨어뜨리는 영역이 존재한다.
- **Overthinking** (Feng et al., 2025): 과도한 추론은 결함 있는 휴리스틱을 증폭하거나, 무관한 세부사항에 고착되는 현상을 유발한다.

본 논문은 이를 8개 모델 × 4개 벤치마크에서 체계적으로 확인했다. 토큰 길이와 정확도의 평균 상관계수는 **r = -0.594**로, 강한 역상관을 보였다. 즉 **길게 쓸수록 오히려 틀릴 확률이 높아진다.**

### 0.3 핵심 가설: "깊이"가 진짜 사고 노력이다

논문이 세우는 가설의 논리 체인:

```
전제 1: 트랜스포머는 L개 레이어를 순차적으로 통과하며 예측을 정제한다.
전제 2: 쉬운 토큰(접속사, 관사, 템플릿 표현)은 초기 레이어에서 이미
        예측이 확정되고, 이후 레이어는 실질적 변경을 가하지 않는다.
전제 3: 어려운 토큰(논리적 판단, 수학적 연산, 핵심 답변)은 깊은 레이어까지
        예측 분포가 계속 수정된다.
가설:   따라서 "몇 번째 레이어에서 예측이 안정화되는가"는
        해당 토큰에 대한 모델의 실제 연산 노력을 반영한다.
```

이를 뒷받침하는 선행 연구:
- **Logit Lens** (nostalgebraist, 2020): 중간 레이어 히든 스테이트에 언임베딩 행렬을 곱하면 해당 레이어 시점의 "잠정적 예측"을 관찰할 수 있다.
- **Tuned Lens** (Belrose et al., 2023): 학습된 어파인 변환을 통해 중간 레이어 예측을 더 정확히 복원할 수 있다.
- **DoLa** (Chuang et al., 2023): 상위/하위 레이어 간 로짓 차이를 이용해 사실 정보 신호를 증폭할 수 있다.

이들 연구가 "중간 레이어의 예측이 의미 있다"는 것을 보여줬다면, 본 논문은 거기서 한 발 더 나아가 "그 예측이 **얼마나 오래 변하는가**가 추론 노력의 척도"라고 주장한다.

### 0.4 방법론: DTR (Deep-Thinking Ratio)

논문은 위 가설을 정량화하기 위해 3단계 측정 파이프라인을 구축한다:

**[1단계] 중간 레이어 예측 분포 추출**

각 생성 스텝 t, 각 레이어 l에서:
- 히든 스테이트 h_{t,l}에 언임베딩 행렬 W_U를 곱해 로짓을 산출
- softmax를 적용하여 확률 분포 p_{t,l}을 얻는다
- 이것이 "레이어 l 시점에서 모델이 생각하는 다음 토큰 예측"이다

**[2단계] 최종 레이어와의 거리 측정**

각 중간 레이어 분포 p_{t,l}과 최종 레이어 분포 p_{t,L} 사이의 Jensen-Shannon Divergence(JSD)를 계산한다. JSD를 선택한 이유:
- 대칭성: JSD(P‖Q) = JSD(Q‖P). KL divergence는 비대칭이라 불안정.
- 유계성: 0 ≤ JSD ≤ 1 (log base 2 기준). 값의 비교가 직관적.
- 실증적 우위: KL divergence(r = -0.698 ~ 0.409)나 cosine similarity(r = 0.172 ~ 0.633) 대비 JSD(r = 0.869 ~ 0.895)가 가장 안정적 상관을 보임.

**[3단계] 수렴 깊이 판정 → DTR 계산**

- 누적 최솟값 D̄_{t,l} = min_{j≤l} D_{t,j}을 추적하여, JSD가 임계값 g 이하로 **처음** 떨어지는 레이어를 "수렴 깊이" c_t로 정의한다.
- 만약 c_t가 전체 레이어의 상위 ρ 비율에 해당하는 영역(deep-thinking regime)에 있으면, 그 토큰은 deep-thinking token이다.
- 시퀀스 전체에서 deep-thinking token의 비율이 DTR이다.

### 0.5 검증: DTR vs 기존 지표

논문은 DTR을 6개 기존 지표와 비교한다. 실험 설계:
- **모델**: GPT-OSS(20B, 120B × low/medium/high reasoning), DeepSeek-R1-70B, Qwen3-30B-Thinking — 총 8개 변종
- **벤치마크**: AIME 2024/2025, HMMT 2025, GPQA-Diamond — 경쟁 수학 + 대학원급 과학 문제
- **디코딩**: 중립적 프롬프트("Please reason step by step"), 질문당 25회 샘플링, 30개 랜덤 시드 평균

결과 (32개 모델-벤치마크 조합에서의 평균 Pearson r):

| 지표 | 평균 r | 양의 상관 비율 |
|------|--------|--------------|
| Token Length | -0.594 | 낮음 (역상관) |
| Reverse Token Length | +0.594 | 사후적 역전 |
| Log Probability | +0.527 | 중간 |
| Negative Perplexity | +0.219 | 불안정 |
| Negative Entropy | +0.571 | 중간 |
| Self-Certainty | +0.605 | 중간~높음 |
| **DTR** | **+0.683** | **30/32 양의 상관** |

> 상세 수치는 섹션 2.8의 공식 기반 표 참조.

DTR만이 32개 중 30개 조합에서 양의 상관을 유지했다. 다른 지표들은 모델이나 벤치마크에 따라 음의 상관으로 뒤집히는 경우가 빈번했다.

### 0.6 하이퍼파라미터 민감도

DTR에는 두 개의 하이퍼파라미터가 있다:
- **g (settling threshold)**: JSD가 이 값 이하면 "수렴"으로 판정
- **ρ (depth fraction)**: 전체 레이어 중 상위 몇 %를 "깊은 영역"으로 정의

실험 결과:
- **g의 영향이 크다**: g = 0.25(관대)이면 shallow 토큰까지 deep으로 분류되어 상관이 약해짐. g = 0.5가 최적, g = 0.75도 유효하지만 필터링이 과도.
- **ρ의 영향은 작다**: ρ = 0.8/0.85/0.9/0.95 모두에서 양의 상관 기울기가 일관되게 유지됨. DTR 값의 절대 범위만 이동.
- **최적 조합**: (g, ρ) = (0.5, 0.85)

### 0.7 응용: Think@n — DTR 기반 test-time scaling

DTR이 정확도와 강하게 상관한다면, 이를 응답 선별에 활용할 수 있다.

**기존 방법 (Self-Consistency, Cons@n)**:
1. 같은 질문에 대해 n개(예: 48개) 응답을 완전히 생성
2. 다수결 투표로 최종 답 선택
3. 문제: 모든 응답을 끝까지 생성해야 하므로 비용이 n배

**Think@n (본 논문 제안)**:
1. n개 응답의 **prefix만** 생성 (ℓ_prefix = 50 토큰)
2. 각 prefix의 DTR을 계산
3. DTR 상위 η%(η = 50%) 응답만 끝까지 생성
4. 선별된 응답으로 다수결 투표

핵심 발견:
- **50토큰이면 충분하다**: prefix 길이를 50, 100, 500, 1000, 2000, 전체로 변화시킨 결과, ℓ_prefix = 50이 오히려 가장 높은 정확도를 기록. 긴 prefix가 더 정확하지 않았다.
- **비용 49% 절감**: 절반의 응답만 완전히 생성하므로 비용이 거의 반으로 줄어듦.
- **정확도 유지/향상**: AIME 2025에서 Cons@48 = 92.7%, Think@48 = 94.7%. 비용은 절반인데 정확도는 오히려 높아짐.

### 0.8 흥미로운 부수 발견: reasoning level과 DTR의 역전

GPT-OSS-120B의 low/medium/high reasoning 설정에서 DTR을 비교한 결과:
- **low reasoning → 높은 DTR**: 짧은 체인, 토큰 당 깊은 사고
- **high reasoning → 낮은 DTR**: 긴 체인, 토큰 당 얕은 사고

해석: high reasoning 모드는 연산을 "깊이"에서 "길이"로 재분배한다. 개별 토큰의 레이어별 연산은 줄어들지만, 더 많은 forward pass(더 많은 토큰)를 생성하여 총 연산량은 늘린다. DTR은 시퀀스 길이로 나누는 비율이므로, 분모가 커지면 자연스럽게 줄어든다.

이는 **DTR을 서로 다른 모델/모드 간에 직접 비교하는 것은 부적절**하다는 것을 의미한다. DTR은 같은 모델, 같은 설정 내에서 응답 간 비교에 사용해야 한다.

### 0.9 논문의 한계 (저자 언급 + 외부 분석)

1. **수렴 ≠ 정확성**: DTR은 모델이 "끝까지 고민했는가"를 측정하지, "그 고민이 올바른 방향이었는가"는 측정하지 못한다. 안정화가 늦다고 반드시 옳은 답은 아니다.
2. **내부 접근 필요**: DTR 계산에는 모든 중간 레이어의 히든 스테이트가 필요하다. API로만 접근 가능한 모델(GPT-4, Claude 등)에서는 직접 측정할 수 없다.
3. **모델 간 비교 불가**: 위 0.8에서 보듯, 서로 다른 모델이나 reasoning 설정 간 DTR 절대값 비교는 무의미하다.
4. **구조적 타당성 미검증**: DTR은 상관(correlation)을 보여줄 뿐, 깊은 레이어 연산이 실제로 논리적/수학적으로 올바른 추론을 수행하는지에 대한 인과적(causal) 설명은 제공하지 않는다.

### 0.10 논문의 핵심 기여 정리

```
[기여 1] 새로운 측정 지표
         DTR = 레이어별 예측 안정화 깊이 기반 추론 노력 측정
         → 토큰 수, 신뢰도 기반 지표를 모두 압도하는 정확도 상관

[기여 2] 실증적 반증
         "길면 좋다"의 통념을 8모델 × 4벤치마크에서 체계적으로 반증
         → r = -0.594로 토큰 길이는 품질의 역지표

[기여 3] 실용적 응용
         Think@n = DTR 기반 조기 필터링으로 test-time scaling 효율화
         → 정확도 유지/향상 + 비용 49% 절감
```

## 2. 수학적 정의 (공식)

### 2.1 기본 설정

L개 트랜스포머 레이어, 히든 차원 d, 어휘 집합 V를 가진 자기회귀 모델 f_θ를 고려한다.

생성 스텝 t에서 각 레이어 l의 히든 스테이트:

```
h_{t,l} ∈ ℝ^d    (l = 1, 2, ..., L)
```

언임베딩 행렬 W_U를 사용해 중간 레이어의 히든 스테이트를 어휘 공간에 투영:

```
z_{t,l} = W_U · h_{t,l}
p_{t,l} = softmax(z_{t,l})
```

여기서 p_{t,l}은 레이어 l에서의 예측 확률 분포, p_{t,L}은 최종 레이어의 출력 분포이다.

### 2.2 레이어 간 거리 측정 (Jensen-Shannon Divergence)

각 생성 스텝 t, 레이어 l에서 중간 레이어 분포와 최종 레이어 분포 사이의 JSD를 계산:

```
D_{t,l} = JSD(p_{t,L} ‖ p_{t,l})
        = H((p_{t,L} + p_{t,l}) / 2) - (1/2)H(p_{t,L}) - (1/2)H(p_{t,l})
```

H(·)는 Shannon 엔트로피이다. D_{t,L} = 0 (자기 자신과의 거리).

- D_{t,l}이 초기 레이어에서 빠르게 0에 수렴 → 모델이 일찍 답을 확정 (shallow)
- D_{t,l}이 깊은 레이어까지 높게 유지 → 모델이 끝까지 고민 (deep-thinking)

### 2.3 Settling Depth (수렴 깊이)

엄격한 수렴 판정을 위해 누적 최솟값을 사용:

```
D̄_{t,l} = min_{j ≤ l} D_{t,j}
```

수렴 깊이 c_t는 D̄_{t,l}이 처음으로 임계값 g 이하로 떨어지는 레이어:

```
c_t = min{ l ∈ {1, ..., L} : D̄_{t,l} ≤ g }
```

### 2.4 Deep-Thinking 영역 정의

깊이 분율 ρ ∈ (0, 1)을 사용하여 "깊은 사고" 영역을 정의:

```
ℒ_deep-thinking = { l : l ≥ ⌈ρ × L⌉ }
```

토큰이 deep-thinking token으로 분류되는 조건:

```
c_t ∈ ℒ_deep-thinking
```

즉, 수렴이 모델의 후반부 레이어에서야 일어나는 토큰이다.

### 2.5 Deep-Thinking Ratio (DTR)

길이 T인 생성 시퀀스 S에 대해:

```
DTR(S) = (1/T) × Σ_{t=1}^{T} 𝟙[c_t ∈ ℒ_deep-thinking]
```

**DTR이 높을수록 = 더 많은 토큰이 깊은 내부 연산을 거침 = 실질적 추론 노력이 높음**

### 2.6 최적 하이퍼파라미터

> 섹션 0.6의 서술적 설명을 수치 참조용으로 정리한 것.

논문 실험에서 확인된 최적 값:

| 파라미터 | 기호 | 최적값 | 의미 |
|----------|------|--------|------|
| Settling 임계값 | g | 0.5 | JSD가 이 값 이하면 "수렴"으로 판정 |
| 깊이 분율 | ρ | 0.85 | 전체 레이어의 상위 15%를 "깊은 영역"으로 정의 |

- g가 너무 낮으면(0.25) → 너무 관대하여 shallow 토큰도 deep으로 분류
- g가 너무 높으면(0.75) → 너무 엄격하여 유의미한 토큰까지 필터링
- ρ는 변화에 비교적 강건하지만, g는 상관계수에 큰 영향

### 2.7 비교 지표 공식 (참고)

DTR과 비교된 기존 지표들:

**평균 로그 확률 (Log Probability)**
```
LogProb(S) = (1/T) × Σ_{t=1}^{T} log p_{t,L}(y_t)
```

**퍼플렉시티 (Perplexity)**
```
PPL(S) = exp(-(1/T) × Σ_{t=1}^{T} log p_{t,L}(y_t))
```

**엔트로피 (Entropy)**
```
Ent(S) = (1/T) × Σ_{t=1}^{T} H(p_{t,L})
```

**Self-Certainty** (균등 분포 u(v) = 1/|V|와의 KL divergence)
```
Self-Certainty(S) = (1/T) × Σ_{t=1}^{T} KL(u ‖ p_{t,L})
```

### 2.8 지표별 정확도 상관계수 비교

| 지표 | 평균 Pearson r | 해석 |
|------|---------------|------|
| Token Length | -0.594 | 길수록 오히려 부정확 |
| Reverse Token Length | +0.594 | 짧을수록 정확 (사후적 역전) |
| Log Probability | +0.527 | 중간 수준 양의 상관 |
| Negative Perplexity | +0.219 | 약한 양의 상관 |
| Negative Entropy | +0.571 | 중간 수준 양의 상관 |
| Self-Certainty | +0.605 | 중간~강한 양의 상관 |
| **DTR (본 논문)** | **+0.683** | **가장 강하고 안정적인 양의 상관** |

### 2.9 DTR 알고리즘 (의사코드)

```
입력: 모델 f_θ (L개 레이어, 언임베딩 행렬 W_U), 프롬프트 x, 임계값 g, 깊이 분율 ρ
출력: DTR(S)

C ← 0                          # deep-thinking 토큰 카운트
S ← ∅                          # 생성된 시퀀스

while y_t ≠ [EOS] do
    y_t ~ p_{t,L}(f_θ(· | x, S))   # 다음 토큰 샘플링
    S ← (S, y_t)

    for l ← 1 to L do
        p_{t,l} ← softmax(W_U · h_{t,l})
        D_{t,l} ← JSD(p_{t,L}, p_{t,l})
    end for

    c_t ← min{ l : min_{j≤l} D_{t,j} ≤ g }    # 수렴 깊이 계산

    if c_t ≥ ⌈ρ × L⌉ then                       # deep-thinking 영역인지 판정
        # 주의: 논문 Algorithm 1의 HTML 렌더링은 ⌈(1-ρ)L⌉로 표시되나,
        #       Equation 5 정의(ℒ = {l : l ≥ ⌈ρL⌉})와 모순됨.
        #       Eq.5 + Figure 3의 의미가 올바르므로 ⌈ρL⌉을 채택.
        C ← C + 1
    end if
end while

return C / |S|                                    # DTR 반환
```

## 3. 직관적 이해

```
[Shallow Token]  → "아, 이건 뻔하지" → 레이어 5에서 이미 답 확정
[Deep-Thinking]  → "잠깐, 다시 생각해봐야..." → 레이어 30까지 분포가 변함

좋은 추론 = Deep-Thinking이 필요한 곳에서 충분히 깊게 사고
나쁜 추론 = Shallow한 곳에서 불필요하게 길게 늘어뜨림
```

논문의 정성적 예시에서도 확인됨:
- **오답 출력**: 27,724 토큰, DTR = 13.9% → 길지만 얕게 사고
- **정답 출력**: 3,725 토큰, DTR = 19.0% → 짧지만 깊게 사고

## 4. 실전 적용 Layer 1 — 시스템 프롬프트 / 에이전트 설정

> 한 번 세팅하면 매 대화마다 자동 적용된다.
> 디렉터가 매번 프롬프트를 수작업할 필요 없이, AI의 행동 기본값을 바꾸는 방법.

### 4.1 claude.md / 시스템 프롬프트에 삽입하는 추론 원칙

아래 블록을 시스템 프롬프트, claude.md, 또는 커스텀 인스트럭션에 복붙하면
해당 세션의 모든 응답에 DTR 기반 추론 원칙이 자동 적용된다.

```markdown
## 추론 원칙 (Deep-Thinking Protocol)

### 깊이 우선 사고
- 자명한 사실에 토큰을 낭비하지 않는다. 확실한 것은 결론만 서술한다.
- 판단이 필요한 분기점을 먼저 식별하고, 그 지점에서만 깊이 사고한다.
- 불확실성이 높은 곳에만 대안 비교를 수행한다.

### Overthinking 방지
- 동일 논점을 3회 이상 반복하지 않는다. 한 번 검증한 사항은 확정 처리한다.
- 이미 결정된 사항을 재논의하지 않는다.
- "~일 수 있다", "~도 고려할 수 있다"를 나열만 하는 양비론을 금지한다.

### 출력 형식
- 결론은 항상 [선택 + 근거] 형태로 제시한다.
- 복잡한 부분만 길고, 단순한 부분은 짧은 비균등 길이 분포가 정상이다.
- 확신이 충분하면 나머지 분석을 생략해도 좋다.
```

### 4.2 역할별 변형 예시

위 기본 블록에 역할 특화 지침을 추가하여 확장할 수 있다.

**코드 에이전트용 추가 블록:**

### 코드 추론 규칙
- 버그 원인 후보는 3개 이하로 좁힌다. 가장 가능성 높은 것부터 검증한다.
- 확인된 원인에 대해서만 수정안을 제시한다. 미확인 가능성은 나열하지 않는다.
- 동작하는 코드를 먼저 제시하고, 설명은 그 뒤에 붙인다.
```

**분석/리서치 에이전트용 추가 블록:**

### 분석 추론 규칙
- 핵심 쟁점을 2~3개로 먼저 압축한 뒤, 각각에 대해서만 깊이 분석한다.
- 최종 답 전에 가장 강력한 반론 1개를 제시하고 반박한다.
- 모든 면을 다루려 하지 않는다. 결정적 포인트에만 집중한다.

- 가장 취약한 부분 1~2곳만 지적한다. 잘 된 부분은 생략해도 된다.
- 각 지적에 [문제 이유 + 대안 1개]를 반드시 함께 제시한다.
- 전체를 훑는 나열식 리뷰를 하지 않는다.
```

## 5. 실전 적용 Layer 2 — AI 출력 품질 평가 체크리스트

> AI가 내놓은 결과물을 검수할 때 쓰는 디렉터의 판별 도구.
> "이 답변이 길기만 한 건지, 깊은 건지"를 빠르게 판단하여
> 재작업 지시를 정확하게 내릴 수 있다.

### 5.1 1차 판별: Overthinking 경고 신호

아래 항목 중 **2개 이상 해당**되면 overthinking 가능성이 높다 → 재생성 또는 수정 지시.

| # | 경고 신호 | 구체적 증상 |
|---|----------|------------|
| 1 | **균일한 길이 분포** | 모든 섹션이 비슷한 분량. 쉬운 부분도 어려운 부분도 같은 길이 |
| 2 | **반복 루프** | 같은 논점을 다른 표현으로 2회 이상 재진술 |
| 3 | **양비론 결론** | "A도 좋고 B도 좋으니 상황에 따라 다르다"로 끝남 |
| 4 | **과잉 전제 나열** | 본론 전에 "먼저 ~를 이해해야 합니다" 류 서론이 전체의 30% 이상 |
| 5 | **자기확인 반복** | "앞서 말했듯이", "다시 정리하면" 등 메타 발화가 3회 이상 |
| 6 | **가능성 나열** | "~일 수 있다", "~도 고려할 만하다"가 결론 없이 4개 이상 |

### 5.2 2차 판별: Deep-Thinking 긍정 신호

아래 항목 중 **3개 이상 해당**되면 실질적으로 깊은 사고가 이루어진 응답.

| # | 긍정 신호 | 구체적 증상 |
|---|----------|------------|
| 1 | **비균등 길이** | 핵심 판단 부분만 길고, 자명한 부분은 1~2줄로 처리 |
| 2 | **명시적 분기점** | "여기서 X와 Y 중 선택해야 하는데"처럼 판단 지점을 식별 |
| 3 | **근거 제시** | 선택에 대해 "왜"를 데이터/논리/사례로 뒷받침 |
| 4 | **반론 검토** | 자신의 결론에 대한 반대 가능성을 언급하고 기각 |
| 5 | **조기 수렴** | 확신이 생긴 시점에서 불필요한 추가 분석을 하지 않음 |
| 6 | **선택 + 이유** | 결론이 "A를 선택한다. 이유는 ~" 형태로 명확 |

### 5.3 재작업 지시 템플릿

판별 결과에 따라 AI에게 내리는 후속 지시:

**Overthinking 감지 시:**
```
결론이 모호하다. 선택지를 하나로 좁히고 이유를 3줄 이내로 밝혀.
반복되는 부분을 모두 제거하고, 핵심 판단 부분만 남겨.
```

**분석이 너무 얕을 때:**
```
결론은 있지만 근거가 부족하다.
왜 다른 선택지가 아닌 이것인지, 가장 강력한 반론 1개를 검토하고 반박해.
```

**길기만 하고 방향이 안 맞을 때:**
```
처음부터 다시 시작해. 이전 응답은 무시해.
핵심 질문은 [X]이다. 이것에만 집중해서 답해.
```

## 6. 실전 적용 Layer 3 — Think@n 기반 워크플로우 설계

> API 호출이나 에이전트 파이프라인에서 후보를 여러 개 뽑고
> 초반부만 보고 필터링하는 작업 프로세스 설계.
> 빈도는 낮지만 비용 절감 효과가 크다.

### 6.1 핵심 원리

논문의 Think@n이 증명한 것:
- 48개 후보 중 **50토큰만 보고** 상위 50%를 골라 나머지를 버리면
- 전체를 다 생성한 뒤 다수결 투표하는 것과 **같거나 더 높은 정확도**
- **비용은 49% 절감**

### 6.2 수동 Think@n: 채팅 인터페이스에서

```
[상황] 중요한 의사결정, 코드 설계 등 정답이 중요한 작업

1단계: 같은 질문을 3~5회 보내서 각각 다른 답변을 받는다
2단계: 각 답변의 첫 3~5문장만 읽는다
        → 문제를 어떻게 프레이밍하는가?
        → 핵심 분기점을 먼저 식별하는가?
        → 결론으로 향하는 방향이 논리적인가?
3단계: 가장 좋은 출발점을 가진 답변 1~2개를 선택한다
4단계: 선택된 답변을 기반으로 후속 작업을 진행한다

→ 나머지 답변은 읽을 필요 없음 (시간 절약)
→ 출발 방향이 좋은 답변이 최종 품질도 높다 (DTR 원리)
```

### 6.3 API/에이전트 파이프라인에서

```python
# Think@n 개념의 API 적용 (의사코드)
# 참고: 실제 API에는 prefix 이어쓰기 파라미터가 없는 경우가 많음.
#       프롬프트 내에 prefix 텍스트를 포함시키거나,
#       assistant prefill 기능을 활용하는 것이 현실적 구현 방법.

# 1단계: n개 후보를 짧은 max_tokens로 생성
candidates = []
for i in range(n):
    response = api.generate(
        prompt=task_prompt,
        max_tokens=100,           # prefix만 생성
        temperature=1.0
    )
    candidates.append(response)

# 2단계: prefix 품질 평가 (사람 또는 평가 모델)
# - 문제 프레이밍이 정확한가?
# - 핵심 분기점을 식별했는가?
# - 논리적 방향성이 있는가?
scored = evaluate_prefixes(candidates)
top_half = select_top(scored, ratio=0.5)

# 3단계: 선택된 후보만 전체 생성
final_responses = []
for candidate in top_half:
    full = api.generate(
        prompt=task_prompt,
        prefix=candidate.text,    # 이어서 생성
        max_tokens=4096
    )
    final_responses.append(full)

# 4단계: 최종 답변 선택 (다수결 또는 최고 품질)
result = majority_vote(final_responses)
```

### 6.4 비용 절감 효과 정리

| 방법 | 정확도 | 비용 | 비고 |
|------|--------|------|------|
| 1회 생성 (Pass@1) | 기준선 | 1x | 운에 의존 |
| 전체 생성 + 다수결 (Cons@n) | 높음 | 48x | 비용 과다 |
| 짧은 것 선택 (Short@n) | 중간 | ~40x | 길이 기반은 불안정 |
| **Think@n (DTR 기반)** | **높음~최고** | **~24x** | **비용 49% 절감** |

## 7. 요약: 기억할 핵심 3가지

```
1. 길게 ≠ 깊게  → 토큰 수는 추론 품질의 역지표 (r = -0.59)
2. 깊이 = 품질  → 실제 판단이 일어나는 곳에 사고를 집중시켜라
3. 조기 필터링  → 시작 50토큰의 방향이 틀리면 끝까지 가봐야 소용없다
```

**DTR 공식 한 줄 요약:**
```
DTR(S) = (깊은 레이어에서야 수렴하는 토큰 수) / (전체 토큰 수)
       → 높을수록 실질적 추론 품질이 높다
```
