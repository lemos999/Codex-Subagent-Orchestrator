> **최종 목적: 보다 빠르고, 빈틈이 없는 게임 개발을 위한 전초기지 — 기획 단계를 완벽하게 완성한다.**

**핵심 용어 정의:**

- **"기획 단계"의 범위:** 게임 DNA 확정(Phase 1)에서 시작하여, 모든 Core 시스템의 수치·규칙·예외가 확정되고 검증 4단계를 통과하여 "코딩 착수 가능"이 선언되는 시점까지. 아이디어 착상 이전(브레인스토밍)과 코딩 착수 이후(프로토타입·QA)는 범위 밖이다.
- **"빈틈 없는 기획":** 이 프레임워크대로 기획을 완료했을 때, 프로그래머가 추가 질문 없이 코딩을 시작할 수 있는 상태.

## 작업 정의

Part A(프레임워크) + Part B(운용 지침)를 **전면 재검토**한 뒤 최적화하라.

**"전면 재검토"란:** 현재 프레임워크의 모든 절차, 판단 기준, 템플릿, 운용 규칙을 하나씩 점검하여 빈틈·비효율·모호함을 찾는 것이다. 현재 상태를 옹호하지 마라. 문제가 없으면 없다고 보고하되, "아마 괜찮을 것이다"로 넘기지 마라.

**"최적화"란:** 보고서에서 끝나지 않는다. 타당성 검증을 통과한 개선 사항은 **프레임워크 수정본**으로 직접 반영하여 제출하라. 보고서만 내고 수정은 안 하는 것은 미완료다.

**Part C(확정 데이터)의 역할:** Part C는 Part A 프레임워크를 실제 프로젝트([1671])에 적용하여 나온 결과물이다. Part C 자체를 수정하는 것이 아니라, "프레임워크가 어디서 잘 작동했고 어디서 실패했는가"를 판단하는 증거로 사용한다. Part C에 빈틈이 보이면, 그것은 프레임워크의 어느 단계가 놓친 것인지 역추적하라.

## 재검토 기준 (5축)

모든 판단은 이 5축으로 귀결된다. 개선안을 제시할 때 반드시 어떤 축에 기여하는지 명시하라.

**5축 간 충돌 시 우선순위:** 빈틈 > 검증력 > 명확성 > 확장성 > 속도. 빈틈을 메우기 위해 절차가 늘어나는 것은 허용한다. 속도를 위해 빈틈을 남기는 것은 허용하지 않는다.

| 기준 | 질문 | 통과 조건 |
|------|------|----------|
| **빈틈** | 이대로 따라가면 기획에 구멍이 생기는 지점이 있는가? | 프레임워크 절차를 빠짐없이 수행했을 때, 코딩 착수에 필요한 정보가 100% 존재한다 |
| **속도** | 불필요한 단계, 중복, 병목이 기획 속도를 늦추는가? | 같은 품질의 기획을 더 적은 단계로 달성할 수 있는 방법이 없다 |
| **명확성** | 이 문서를 읽고 즉시 실행 가능한가? | 프레임워크를 처음 접한 에이전트가 추가 질문 없이 Phase 1부터 실행할 수 있다 |
| **확장성** | 다른 장르·규모에도 적용 가능한가? | 생존 샌드박스 외 3개 이상 장르에서 절차가 깨지지 않는다 |
| **검증력** | 기획 완료 후 "코딩 착수 가능"이라 확신할 수 있는가? | 검증 4단계를 통과한 기획서에서 구현 중 "이건 기획에 없는데?"가 발생하지 않는다 |

## 검토 항목

1. **프레임워크 구조(Part A)** — Phase 순서가 최적인가, 빠진 Phase가 있는가, 판단 기준이 모호한 곳은 없는가, 22개 템플릿이 과잉이거나 부족하지 않은가, 핀셋 인터뷰가 의사결정을 가속하는가
2. **프로세스 흐름(Part A+B)** — DNA→우선순위→핀셋→검증→문서화 파이프라인의 병목, 파트 간 교차 검증 시점이 적절한가, "코딩 착수 가능" 선언의 게이트 기준이 명확한가
3. **실전 역추적(Part C 기반)** — Part C의 cross-part-contradictions(C-7)에서 발견된 모순의 패턴을 분석하고, 프레임워크 단계에서 사전 차단할 수 있는 체크포인트를 제안하라
4. **범용성** — 현재 [1671](하드코어 생존 샌드박스)에 특화된 부분을 식별하고, 장르 무관 보편 원칙과 장르별 분기가 필요한 원칙을 구분하라
5. **추가 제안** — 현재 프레임워크에 없는데 있어야 하는 것을 제안하라

## 타당성 검증 기준

개선안이나 추가 제안은 아래 기준을 충족해야 적용된다. 충족하지 못하면 보고서에 기록만 하고 적용하지 않는다.

| 등급 | 조건 | 처리 |
|------|------|------|
| **[필수]** | 이것이 없으면 5축 중 하나 이상에서 실패한다는 구체적 시나리오를 제시할 수 있다 | 즉시 적용 |
| **[권고]** | 5축 중 하나 이상이 측정 가능한 수준으로 개선된다는 근거가 있다 | 적용 (근거 명시) |
| **[제안]** | 개선 가능성은 있으나 실전 검증이 필요하다 | 보고서에만 기록. 적용 보류 |
| **미달** | 근거 없는 일반론, "~하면 좋을 것이다" 수준 | 기록하지 않는다 |

**절차를 늘리는 제안은 추가 기준이 적용된다:** 늘어나는 단계 수 × 소요 시간이 줄어드는 빈틈의 심각도보다 작아야 정당화된다. 절차 증가 자체가 비용이다.

## 불변 원칙

이 5개 원칙은 최적화 과정에서 절대 훼손하지 않는다. 오히려 더 강화하는 방향으로 개선한다. 개선안이 이 원칙과 충돌하면, 개선안을 버린다.

1. **재미가 최우선.** 모든 시스템은 "왜 재미있는가?"로 정당화되어야 한다. 재미 없으면 완성도는 무의미.
2. **모호한 결정은 결정이 아니다.** 수치·규칙·조건 중 적합한 형태로 명시한다. "빠른 전투" ✗ → "딜레이 0.3초" ✓. 기획서에 명시적 정의가 없는 항목은 미완료다.
3. **What만 정의, How는 개발자 판단.** 엔진 중립 언어만 사용. 구현 방식을 강요하지 않는다.
4. **더 복잡하게가 아니라, 더 명확하고 빠르게.** 절차를 추가하려면 그만큼 빈틈이 줄어든다는 증거가 있어야 한다.
5. **Part C는 수정 불가.** 프레임워크 평가의 증거로만 활용한다.

## A-5. 기획서 작성 표준

| 유형 | 예시 | 판정 |
|------|------|------|
| 소설 언어 | "시야에 들어오면 경계한다" | ❌ 기준 불명확 |
| 엔진 API | `Raycast(15m, FOV 120°)` | ❌ 구현 강요 |
| **기획서 표준** | "전방 120°, 15m 이내, 지형지물 미차단 시 탐지" | ✅ |

**엔진 용어 순화:** Vector3.Distance → "직선거리", Trigger Collider → "반경 내 진입 판정", FSM.State → "현재 행동 상태"

1. 비개발자가 읽어서 이해 가능한가?
2. 프로그래머에게 구현 자유를 막지 않았나?

### Phase 1: 게임 DNA 확정

사용자에게 아래 5가지를 순서대로 확인한다.

1. **핵심 감정** — 플레이어가 느껴야 할 감정 1가지
2. **코어 루프** — 마이크로(30초~3분) / 미들(5~30분) / 매크로(1시간~세션)
3. **장르 컨벤션** — 해당 장르에서 당연히 기대되는 요소 + 의도적 제거 항목
4. **고유 차별점(USP)** — "[장르]인데 [다른 점] 때문에 [새로운 경험]"
5. **타겟 플레이어** — 연령대, 세션 길이, 실패 허용도, 플레이 환경

DNA 5항목이 모두 확정되면 Phase 2로 진행.

### 핵심 이론

- **MDA 프레임워크** — Mechanics → Dynamics → Aesthetics. 8가지 Aesthetic 중 1~2개만 목표.
- **플로우 이론** — 도전 ≈ 능력일 때 몰입. 도전 >> 능력 = 좌절, 능력 >> 도전 = 지루함.
- **보상 심리학** — Variable Ratio(가변 비율)가 가장 강한 동기. 기본 보상(예측 가능) + 추가 보상(가변) = 최적.
- **피드백 루프** — 긍정(강해질수록 강해짐) + 부정(강해질수록 저항) 혼합이 최적.

### 재미 4문항 적용법

<!-- [P2-변경] A-8 중복 4문항 제거, A-3 참조로 대체. 명확성/속도 축 강화. A-3과 A-8 간 동일 정의 이중 기술 해소. -->

<!-- [변경] F-U1: 복수 장르 예시 병렬 배치. 확장성 축 강화. 생존 게임 전용 인상 해소. -->
**적용 예시 (생존):** 온돌 시스템 — 압박(한파→사망) ✅ / 대응(연료 관리, 외출 타이밍) ✅ / 피드백(체온 아이콘, 입김) ✅ / 성장(효율적 관리 체화) ✅ → Core

**적용 예시 (액션RPG):** 패링 시스템 — 압박(적 공격) ✅ / 대응(타이밍 맞춰 패링) ✅ / 피드백(패링 성공 이펙트+경직) ✅ / 성장(패링 타이밍 체화) ✅ → Core

**적용 예시 (덱빌더):** 카드 드로우 — 압박(마나 부족) ✅ / 대응(카드 선택) ✅ / 피드백(즉각 효과 연출) ✅ / 성장(덱 구성 최적화) ✅ → Core

**환경 스토리텔링 오브젝트:**
압박 ❌ / 대응 △ / 피드백 △ / 성장 ❌ → Support

#### 판정

| 통과 | 등급 | 처리 |
|------|------|------|
| 4개 | Core | 최우선 설계 |
| 3개 | Core 후보 | 약한 1개 강화 |
| 2개 | Support | Core 완성 후 연결 |
| 1개↓ | 재검토 | "왜 만드나?" |

#### 재미 없는 시스템의 흔한 패턴

1. 압박 없음 → 자원 희소성 추가
2. 대응 없음 → 선택지 도입
3. 피드백 없음 → 이펙트·숫자 팝업
4. 성장 없음 → 점진적 복잡성

**장르별 일반적 Core 시스템:**

> 이 테이블은 대표 예시이며, 게임에 맞는 장르를 추가할 수 있다.

| 장르 | Core 후보 |
|------|----------|
| 공포/호러 | 이동 → AI → 환경 → 서사 → UI(사운드) |
| 농장/생활 시뮬 | 환경(시간/계절) → 크래프팅 → NPC관계 → 경제 → 건설 |
| 덱빌더 | 덱/카드 → 전투(턴제) → 성장 → 맵(노드) |
| 레이싱 | 이동·물리 → 탈것 → 스코어링 → 맵 |
| 로그라이크 | 전투 → 성장(메타) → 맵(절차생성) → 인벤토리 |
| 리듬 게임 | 스코어링 → 난이도 → UI |

## ✅ 데미지 공식
```
최종 데미지 = Max(1, (무기 공격력 × 피로도 배율) - 타겟 방어력)
```
- 피로도 100% → 배율 1.0 / 10% 미만 → 기하급수적 감소 / 최소 = 1(팅김)

## ✅ 붕괴 시스템
- 피격 시 붕괴 공격력만큼 게이지↓ → 0 = **스턴 3초**
- 회복: 0 도달 또는 5초 미피격 → 즉시 100% 리셋
- **면역:** 스턴 중(3초) + 직후(2초) = 붕괴 면역 (무한 스턴 차단)

# C-6. 어그로/습격 시스템
# ─────────────────────────────────────

> 최종 갱신: 2026-03-02
> 인벤토리(요리), NPC(적 스폰), 레벨(거점 방어)에 걸쳐 있는 시스템.

## ✅ 시스템 개요
살기 위한 최소 행위(요리/정수)가 다가올 거대한 위협의 원인이 되는 딜레마 시스템.
솥에서 음식을 끓이는 순간, 연기와 냄새가 적을 불러온다.

## ✅ 트리거 조건

| 조건 | 내용 |
|------|------|
| 발동 시점 | 아궁이+솥에서 [끓이기] 버튼을 누르는 순간 |
| 데이터 축적 | 요리/정수 **성공** 시 `Success_Aggro_Value` 1 누적 |
| 타이머 시작 | 첫 성공 시 **3일(인게임 72시간) 카운트다운** 시작 |

## ✅ 확률 로직

| 항목 | 수치 |
|------|------|
| 첫 번째 요리 성공 시 적 등장 확률 | **44.44%** |
| 이후 요리 성공마다 증가 | **+4%** |
| 최대 확률 | **100%** |
| 확률 초기화 조건 | 3일 타이머 만료 시 리셋 |

## ✅ 적 스폰 로직

| 항목 | 내용 |
|------|------|
| 스폰 타이밍 | 72시간 타이머 0일 때, 확률 판정 통과 시 |
| 스폰 위치 | 솥이 있는 아궁이의 좌표 |
| 스폰 대상 | 짐승/도적 떼/유민/관군 중 랜덤 |
| 스폰 규모 | 생존 일수 기반 스케일링 |

## ✅ 스폰 규모 스케일링

| 생존 일수 | 기본 스폰 수 | 적 조합 |
|----------|-------------|---------|
| 1~10일 | 5명 | 단일 종족 |
| 11~20일 | 7명 | 단일 + 대장 30% |
| 21~30일 | 9명 | 혼합 |
| 31~50일 | 12명 | 혼합 + 대장 확정 |
| 51일+ | **15명 (캡)** | 혼합 + 대장 2 + 호랑이 5% |

## ✅ 거점 방어 연동

| 항목 | 내용 |
|------|------|
| 방어 수단 | 덫, 바리케이드, 지형 활용 |
| 엔드 콘텐츠 주기 | 3일마다 반복되는 방어전 |
| 은신 대안 | 은신 시 적 AI → 거점 문 부수기 → 뒤주 파괴 → 강아지 공격 |

## ✅ 날씨 예외 처리

| 상황 | 솥 타이머 | 어그로 타이머 |
|------|----------|-------------|
| 정상 요리 | 실시간 진행 | 실시간 진행 |
| 붉은 비로 불 꺼짐 | **일시정지** | **계속 진행** |
| 불 재점화 후 | 이어서 재개 | 영향 없음 |

## ✅ 요리별 대기 시간

| 요리 | 대기 시간 |
|------|---------|
| 물 정수 | 10초 |
| 구황작물 요리 | 20초 |
| 탕약 | 45초 |

## ✅ 솥 내구도

| 솥 | 용량 | 내구도 | 파괴 시점 |
|----|------|--------|----------|
| 이가 빠진 옹기 | 물 1회, 음식 1개 | 3회 | 요리 100% 완료 시 |
| 무쇠 가마솥 | 물 3회, 대량 | 무한 | — |

## ✅ 어그로 타이머 UI
- **표시 없음** — 타이머 수치 미표시
- 대체 힌트: 강아지 불안 행동 + 망루지기 대사
- 발동 직전(30초 전): "어디선가 발자국 소리가 들린다" 자막 + 강아지 Alert

## ✅ 복수 아궁이 어그로
- 거점 내 최대 **1개**만 활성화 가능
- 야영 모닥불은 어그로 시스템 **미연동** (연기량 적음)
- 야영 모닥불 용도: 체온 회복 + 저장 전용. 요리 불가
