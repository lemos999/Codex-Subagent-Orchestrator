# Claude Opus 4.6 작업 절차

대상 문서: `game-design-director/game-design-director-integrated.md`  
현재 상태: Pass 3 완료, `90/100` 도달  
목표: 현재 버전을 무너뜨리지 않고, 같은 방식으로 더 높은 완성도에 도달

---

## 1. 작업 목표

당신의 임무는 `game-design-director-integrated.md`가 **자기 문서가 선언한 목표**에 더 아름답고, 더 엄밀하고, 더 실전적으로 도달하도록 만드는 것이다.

핵심 원칙은 다음과 같다.

- 이미 해결된 문제를 다시 흐리지 말 것
- 문서를 더 길게 만드는 것보다 더 강하게 만드는 것을 우선할 것
- 근거 없는 장식, 일반론, 추상적 미사여구는 추가하지 말 것
- `Part C`는 절대 수정하지 말고 증거로만 사용할 것

## 4. 작업 방식

이번 작업도 이전과 같은 구조를 따른다.

### Step 1. 현재 버전 재평가

현재 `90/100` 판정을 그대로 받아들이지 말고 다시 검토한다.

특히 아래를 본다.

- Pass 1, 2, 3에서 남긴 잔여 한계가 아직 유효한지
- 이미 들어간 변경이 서로 충돌하거나 중복되는지
- "좋은 추가"가 아니라 "좋은 제거/정리"가 더 맞는 지점이 있는지

### Step 2. 남은 고가치 이슈만 선별

아래 조건을 만족하는 것만 수정 후보로 올린다.

- 5축 중 하나 이상을 실제로 올린다
- 이전 패스보다 더 나은 근거가 있다
- 문서 부피 증가보다 효과가 크다
- 기존 구조를 망가뜨리지 않는다

다음 유형은 우선순위가 낮다.

- 보기 좋은 표현 치장
- 근거 없는 템플릿 추가
- 장르 일반론 확대
- 실전 검증 없이 넣는 선언적 규칙

### Step 3. 최소 변경으로 최대 효과

변경은 가능한 한 작게 넣는다.

좋은 변경의 예:

- 기존 섹션 사이의 역할 분담을 명확하게 함
- 전이 규칙, 게이트, 검증 우선순위를 더 선명하게 함
- 닫힌 목록처럼 보이는 부분에 대표성/확장성 규칙을 짧게 추가
- 프로젝트 특화 검증 훅을 넣되 범용 프레임워크를 오염시키지 않음

나쁜 변경의 예:

- 새 대분류 섹션을 크게 추가함
- 새 템플릿을 대량으로 늘림
- A와 B에서 같은 내용을 다시 반복함
- Part C를 암묵적으로 재해석하면서 사실상 수정함

---

## 5. 판단 기준

수정 여부는 아래 순서로 판단한다.

1. **빈틈**을 줄이는가
2. **검증력**을 높이는가
3. **명확성**을 높이는가
4. **확장성**을 높이는가
5. **속도**를 높이는가

속도를 높이기 위해 빈틈이나 검증력을 깎는 수정은 하지 않는다.

---

## 6. 권장 작업 루프

아래 루프로 진행한다.

### Pass A. Reviewer 관점

현재 문서에서 아래를 먼저 적출한다.

- 아직 남은 구조적 빈틈
- 과잉 규칙 또는 중복
- 애매한 게이트
- 장르 편향 또는 닫힌 구조로 읽힐 수 있는 부분
- 프로젝트 특화 규칙이 범용 프레임워크를 오염시키는 지점

### Pass B. Implementer 관점

위에서 고른 상위 이슈만 실제 문서에 반영한다.

규칙:

- 큰 리라이트보다 국소 수정 우선
- 기존 `[변경]`, `[P1-변경]`, `[P2-변경]`, `[P3-변경]` 태그는 보존
- 새 변경은 일관된 새 태그로 남겨도 되고, 별도 리뷰 문서에서 추적해도 됨

### Pass C. Reviewer 관점 재검증

반드시 다시 검토한다.

확인 항목:

- 수정이 Part A/B에만 들어갔는가
- 새로운 중복이나 모순이 생기지 않았는가
- 이전보다 실질적으로 좋아졌는가
- Part C 해시가 유지되는가

문제가 있으면 작은 fixer 패스로만 바로잡고 다시 검토한다.

## 9. 특히 조심할 것

- 이미 들어간 개선을 무심코 제거하지 말 것
- Pass 3에서 확보한 `확장성`, `검증력`, `속도` 개선 포인트를 되돌리지 말 것
- "더 좋아 보이는 새 체계"를 만들기 위해 기존 체계를 뒤집지 말 것
- 문서의 힘은 "짧아서"가 아니라 "판단이 선명해서" 생긴다는 점을 기억할 것

## 10. Opus에게 기대하는 최종 결과

좋은 결과는 아래와 같다.

- 기존 90점 구조를 보존한다
- 정말 남은 고가치 문제만 건드린다
- 문서를 더 무겁게 하지 않고 더 믿을 수 있게 만든다
- 다음 작업자나 프로그래머가 읽었을 때 질문 수가 더 줄어든다
- Part C는 완전히 동일하게 유지된다

나쁜 결과는 아래와 같다.

- 문서가 더 길어졌지만 더 선명해지지 않는다
- 범용성과 프로젝트 특화의 경계를 흐린다
- 이전 패스의 판단 근거를 무시하고 다시 흔든다
- 새로운 템플릿, 새로운 섹션, 새로운 규칙을 과도하게 늘린다
