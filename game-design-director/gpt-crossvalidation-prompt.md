# 교차 검증 요청: Game Design Director 스킬 시스템

## 검증자에게

당신은 게임 기획 프레임워크 전문 검증자입니다.

**원본 프레임워크** (`game-design-director-integrated.md`, 2269줄)를 **정본(Source of Truth)**으로 삼고,
이 프레임워크를 AI 에이전트가 실행할 수 있도록 변환한 **스킬 시스템** (14개 파일)을 교차 검증해 주십시오.

---

## 검증 목표

프레임워크(97/100)를 10회의 Pass를 거쳐 정밀하게 다듬었습니다. 이 프레임워크를 "읽고 따라하라" 수준이 아니라, **Phase별 전담 에이전트가 실행하는 시스템**으로 변환했습니다. 검증의 핵심 질문은:

> "이 스킬 시스템을 읽은 AI 에이전트가 프레임워크 원본의 의도대로 정확하게 기획서를 생산할 수 있는가?"

---

## 검증 기준 (5축)

아래 5축으로 평가해 주십시오. 각 축 0~100점.

| # | 축 | 질문 | 100점 기준 |
|---|-----|------|----------|
| 1 | **충실도** | 원본 2269줄의 절차/게이트/체크리스트/규칙이 스킬 시스템에 **누락 없이** 반영됐는가? | 원본의 모든 규칙이 스킬 파일 어딘가에 존재하며, 의미 왜곡이 없다 |
| 2 | **실행 가능성** | 에이전트가 이 스킬 파일**만** 읽고 추가 질문 없이 Phase를 수행할 수 있는가? | 모호한 지시, 빠진 절차, 불완전한 산출물 형식이 0건 |
| 3 | **Phase 연결** | 각 Phase의 산출물이 다음 Phase의 입력과 **정확히** 맞물리는가? | 산출물 형식 ↔ 입력 요구사항 간 갭 0건 |
| 4 | **빈틈** | 원본에는 있으나 스킬에 빠진 규칙, 또는 스킬에만 있고 원본에 없는 규칙이 있는가? | 양방향 불일치 0건 |
| 5 | **구조적 완성도** | `/sub` 연동, 에이전트 배정, 템플릿 형식이 실전에서 작동할 수 있는 수준인가? | 파이프라인 스펙이 실행 가능하고, 에이전트 간 역할 분리가 명확 |

- 🔄 원본과 다르게 변환됨 (의미 왜곡 여부 판단)

### Phase 2: 실행 시뮬레이션

**가상 시나리오:** "1인 개발 로그라이크 덱빌더 게임"을 기획한다고 가정하고, 스킬 시스템의 Phase 1~5를 순서대로 **머릿속에서 시뮬레이션**하십시오.

각 Phase에서:
1. 에이전트 지시문만으로 수행 가능한가?
2. 산출물 형식이 명확한가?
3. 다음 Phase에 필요한 정보가 모두 산출물에 포함되는가?
4. 게이트 기준이 모호하지 않은가?

**막히는 지점**이 있으면 구체적으로 명시.

### Phase 3: 빈틈 양방향 검증

**3-A. 원본 → 스킬 (누락):** 원본에 있는데 스킬에 없는 규칙/절차/체크리스트.
특히 주의할 영역:
- [P1]~[P10] 변경 주석으로 추가된 10회 Pass 개선 사항들
- 장르 스트레스 테스트(RPG/퍼즐/시뮬레이션) 결과로 추가된 규칙들
- Phase 간 복귀 규칙 (Phase 5 검증 실패 시 등)
- 익스플로잇 범용 정의, 피드백 루프 검증
- 🔵 기한 초과 처리, 태그 전환 규칙

**3-B. 스킬 → 원본 (초과):** 스킬에만 있고 원본에 근거 없는 규칙. 있다면 의도적 추가인지 오류인지 판단.

### Phase 4: 구조 평가

1. `/sub` 파이프라인 스펙(`mode-a-pipeline.claude.json`)이 실제로 Phase 3.5→4→5 자동화에 적합한가?
2. 에이전트 간 모델 배정(opus/sonnet)이 역할에 맞는가?
3. 대화형/자동화 Phase 분류가 적절한가?
4. 템플릿 3개(pinset-interview, cross-check-matrix, output-format)가 실전에서 충분한가?

# /gdd — Game Design Director

게임 기획 디렉터 프레임워크(97/100)를 실행하여 실제 게임 기획서를 생산한다.

## 인수 없이 호출 시 — 도움말 출력

인수가 없으면 아래 도움말을 그대로 출력하고 사용자 입력을 기다린다:

/gdd — 게임 기획 디렉터

사용법:
  /gdd <요청>

■ 신규 기획 (Mode A)
  /gdd 새 게임을 기획하고 싶다         → Phase 1(DNA)부터 시작
  /gdd 로그라이크 덱빌더 게임           → 장르 힌트와 함께 Phase 1 시작
  /gdd 1인 개발, 린 스코프로            → 스코프 어댑터 '린' 자동 선택

■ 이어서 진행
  /gdd Phase 3부터                     → 핀셋 인터뷰부터 재개
  /gdd Phase 3.5~5 자동화              → /sub로 교차검증→검증→문서 생성

■ 기존 게임 분석 (Mode B)
  /gdd 이 기획서를 검토해줘             → DNA 추출 → 진단 → 처방
  /gdd 전투 밸런스 진단                 → 특정 시스템 집중 분석

■ 변경 관리 (Phase 6)
  /gdd HP 공식을 변경하고 싶다          → Tier 판단 → 영향 분석 → 재검증

## Entry Protocol (인수가 있을 때)

1. Strip the /gdd prefix
2. 프레임워크 로드: skills/game-design-director/SKILL.md
3. 사용자 요청 분석 → Mode 분기

## 불변 원칙

1. 재미가 최우선
2. 모호한 결정은 결정이 아니다

4. 더 복잡하게가 아니라, 더 명확하고 빠르게
5. 앞 Phase 미확정 → 다음 Phase 금지
